# 21-统一能力模型

## Purpose
统一 CLAW 的核心对象和能力抽象，为后续所有规范提供一致的类型语言。

## Scope
本文件定义对象职责和 owner，不展开完整字段 schema。

## Actors / Owners
- Owner: Architecture
- Readers: 全部实现者

## Inputs / Outputs
- Inputs: 术语表、AgentOS 集成规范、任务图规范
- Outputs: 核心对象清单、能力分类、owner mapping

## Core Concepts
核心对象 owner：

| Type | Role | Owner Spec |
|---|---|---|
| `AgentCapability` | 统一能力声明 | 20 |
| `AgentAdapter` | 统一适配接口 | 20 |
| `Session` | 用户级执行容器 | 25 |
| `TaskNode` | 最小可调度节点 | 22 |
| `WorkerRun` | 一次实际执行实例 | 25 |
| `ContextSnapshot` | 完整上下文快照 | 23 |
| `ContextDiff` | 增量同步对象 | 23 |
| `EventEnvelope` | 统一事件信封 | 24 |
| `ReplayDocument` | 由事件生成的回放文档 | 32 |
| `AnalysisReport` | 分析产物 | 32 |
| `SpecAsset` | 可版本化规范资产 | 31 |
| `RuntimeAsset` | 可追溯运行资产 | 26 |

能力模型分层：

| Layer | Examples |
|---|---|
| Core | session、input、status、event stream |
| Orchestration | task dispatch、interrupt、subagent |
| Evidence | file change、permission signal、structured result |
| Extension | agent-specific raw hook、provider metadata |

## Behavior / Flow
- 上层模块只能依赖 `Core + Orchestration + Evidence`。
- `Extension` 不允许成为主流程硬依赖，只能作为增强能力。
- 新对象必须先声明 owner，再进入实现细化。

## Interfaces / Types
能力声明建议结构：

```python
class AgentCapability(TypedDict):
    name: str
    level: Literal["core", "orchestration", "evidence", "extension"]
    supported: bool
    notes: str | None
```

## Failure Modes
- 如果对象 owner 不明确，后续规范会重复定义。
- 如果 extension 进入主流程，系统将失去跨 AgentOS 的可迁移性。

## Observability
- 对象创建、变更和消费的关键节点都需要能映射到对应的 owner spec。

## Open Questions / ADR Links
- 若未来引入 `Policy` 作为独立对象，建议由新 ADR 定义 owner。

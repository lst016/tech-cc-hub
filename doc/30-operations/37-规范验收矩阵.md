# 37-规范验收矩阵

## Purpose
为 CLAW 的核心规范建立统一验收标准，确保文档能被实现、验证和持续维护。

## Scope
本文件定义“每份 spec 如何被验收”，而不是定义产品功能本身。

## Actors / Owners
- Owner: Architecture
- Readers: 实现者、评审者、测试与验收负责人

## Inputs / Outputs
- Inputs: L1/L2/L3 各规范
- Outputs: 验收矩阵、最小验证方式、证据要求

## Core Concepts
- `Acceptance Evidence`: 证明规范被实现或可实现的证据。
- `Traceability`: 一个能力能从产品定义追到事件和产物。
- `Review Gate`: 文档是否达到可进入实现阶段的门槛。

## Behavior / Flow
验收维度：

1. `结构完整`
   文档是否有 Purpose/Scope/Interfaces/Observability 等核心段落。
2. `对象闭环`
   涉及的核心对象是否有 owner、有最小 schema、有上下游链接。
3. `运行闭环`
   是否能串到 Session、Task、Worker、Event、Replay 主链路。
4. `证据闭环`
   是否说明需要哪些事件、产物或日志来证明实现正确。
5. `渐进实现`
   是否允许先最小实现、后增强，而不是一次到位。

## Interfaces / Types
规范验收矩阵：

| Spec | Primary Check | Required Evidence | Ready For Implementation |
|---|---|---|---|
| `00-产品定义` | 边界清晰 | 产品边界、非目标、主链路 | yes |
| `04-问题定义与成功指标` | 价值清晰 | 目标用户、JTBD、成功指标 | yes |
| `10-11 C4 图` | 容器清晰 | Context/Container 图可读 | yes |
| `20-AgentOS集成规范` | 适配边界清晰 | Adapter contract、职责边界 | yes |
| `22-任务图与递归拆分规范` | 调度可执行 | TaskNode、budget、状态 | yes |
| `23-上下文同步与合并规范` | 回写可解释 | snapshot/diff/conflict 流程 | near-ready |
| `24-事件模型与可观测规范` | 回放链路完整 | EventEnvelope、事件族、边界事件 | yes |
| `25-会话与状态机规范` | 状态可驱动 UI/调度 | state enums、转移关系 | yes |
| `26-存储与Markdown产物规范` | 资产分层清晰 | spec/runtime/artifact 分层 | near-ready |
| `29-AgentOS能力映射矩阵` | 双 Agent 可共存 | capability mapping、fallback | yes |
| `32-回放与分析报告规范` | 证据可读 | replay/report 结构、回链要求 | near-ready |
| `36-端到端场景样例` | 最小闭环验证 | 场景时间线、关键事件 | yes |
| `34-MVP切片与迭代路线图` | 交付可验证 | 切片目标、价值假设、成功指标 | yes |

验收状态建议：
- `draft`: 只有方向，没有实现入口
- `near-ready`: 主结构已清晰，但还缺字段或样例
- `ready`: 可直接拆实现任务

## Failure Modes
- 没有验收矩阵时，文档容易一直“看起来不错”，但没人知道何时可开工。
- 如果只按篇幅验收，会把低质量长文误判为完成。

## Observability
- 每份准备进入实现的 spec 都应能提供:
  - 一个 owner
  - 一组关键对象
  - 一组最少事件
  - 一份可验证 evidence

## Open Questions / ADR Links
- 实现开始后，可把本矩阵接入项目管理看板，形成文档完成度视图。

# 33-冲突处理与人工介入规范

## Purpose
定义 CLAW 如何处理并行结果冲突，以及用户如何以一等方式介入运行时。

## Scope
本文件覆盖冲突类型、人工接管动作、决策记录和回放要求。

## Actors / Owners
- Owner: Core Runtime + Frontend
- Readers: 调度、前端、分析实现者

## Inputs / Outputs
- Inputs: ContextDiff、merge attempts、permission requests、user decisions
- Outputs: ConflictRecord、decision events、resolved state

## Core Concepts
- `ConflictRecord`: 冲突实体。
- `Human Intervention`: 用户对系统状态的显式修正或决策。
- `Resolution Strategy`: 冲突的处理方式。

## Behavior / Flow
冲突分型：

| Conflict Type | Example |
|---|---|
| `file_conflict` | 多个 Worker 修改同一路径 |
| `context_conflict` | shared_state 回写不一致 |
| `policy_conflict` | 子任务尝试越过预算或路径限制 |
| `version_conflict` | 基于过期 snapshot 回写 |
| `decision_conflict` | 自动策略和人工指令相冲突 |

人工介入动作：
- approve
- reject
- reroute
- revise task
- narrow scope
- retry with new spec

## Interfaces / Types
`ConflictRecord` 至少包含：
- `conflict_id`
- `conflict_type`
- `detected_at`
- `related_session_id`
- `related_task_ids`
- `evidence_refs`
- `resolution_status`

## Failure Modes
- 如果人工介入不形成事件，后续回放无法解释“为什么系统突然改变方向”。
- 如果所有冲突都自动解决，系统会失去企业可控性。

## Observability
- 所有人工介入必须有独立事件和关联证据。
- 记录冲突发现率、解决时延、自动解决与人工解决占比。

## Open Questions / ADR Links
- 某些冲突是否允许“策略自动裁决”，留待后续 ADR。

# 36-端到端场景样例

## Purpose
用小而完整的样例验证 CLAW 文档体系不是停留在概念层，而是能串起真实产品闭环。

## Scope
本文件提供两个标准场景：
- 单 Agent 单任务
- Hub + 多 Worker + 人工介入 + 回放生成
- 同类任务的调优前后对比

## Actors / Owners
- Owner: Architecture + Product
- Readers: 实现者、评审者、后续写测试和 demo 的成员

## Inputs / Outputs
- Inputs: 用户目标、Task Graph、AgentOS 响应、事件、人工决策
- Outputs: 场景时间线、关键对象、预期产物

## Core Concepts
- 场景样例是“最小闭环证明”，不是完整业务剧本。
- 每个场景都要明确：
  - 输入 spec
  - 任务节点
  - 关键事件
  - 产物
  - 验收点

## Behavior / Flow
### 场景 A: 单 Agent 单任务

目标：
用户要求“生成一个登录 API 原型”，使用 Claude Code 执行，无递归拆分。

流程：
1. 用户在 Chat 创建 `Session`。
2. CLAW 创建 root `TaskNode`。
3. Hub 直接创建一个 `WorkerRun` 并分配 Claude Code。
4. 适配器启动底层会话，持续归一事件。
5. 执行结束后生成 `ResultSummary`。
6. 事件流写入存储，生成 `ReplayDocument` 和 `AnalysisReport`。

关键对象：
- `Session`: 1 个
- `TaskNode`: 1 个
- `WorkerRun`: 1 个
- `EventEnvelope`: 至少包括 session start、task assign、worker start、agent tool、worker complete、artifact generate

预期产物：
- 一份回放文档
- 一份分析报告
- 一组事件和状态快照

### 场景 B: Hub + 多 Worker + 人工介入

目标：
用户要求“实现用户系统 + 订单系统 + 集成测试”，Hub 拆成多个任务，使用 Claude Code 和 Codex 混合执行。

流程：
1. 用户在 Task Graph 中确认 root goal。
2. Hub 生成 root `TaskNode` 并拆成:
   - 用户系统
   - 订单系统
   - 集成测试
3. Hub 根据能力矩阵分配：
   - Claude Code 负责用户系统
   - Codex 负责订单系统
   - 集成测试等待依赖完成
4. 两个 `WorkerRun` 并行执行，持续产生事件。
5. 订单系统在文件冲突处触发 `ConflictRecord`。
6. 用户在 UI 中介入，选择保留用户系统版本并要求订单系统重试。
7. Hub 生成新的 `WorkerRun` attempt，并完成合并。
8. Session 结束后生成回放和分析。

关键事件：
- `task.split`
- `worker.start`
- `agent.tool.*`
- `context.conflict`
- `human.revise`
- `worker.retry`
- `task.merge`
- `artifact.replay_generated`

预期产物：
- 可见的任务图演化
- 冲突记录
- 含人工介入片段的回放
- 能指出返工原因的分析报告

### 场景 C: 调优前后对比

目标：
证明 CLAW 的差异价值不只是“能执行任务”，而是“能把一次任务经验沉淀为下一次更稳的执行方法”。

流程：
1. 用户第一次执行“生成带权限控制的后台 API”任务，未绑定成熟 workflow / skill。
2. 系统完成执行，但回放显示：
   - 人工多次补充约束
   - 权限策略表达不一致
   - 返工集中在接口规范和权限边界
3. 用户基于分析报告，把经验整理成：
   - 一个 `workflow`
   - 一个 `skill`
   - 一个 `policy`
4. 用户第二次执行相似任务，Session 明确引用这些 `SpecAsset`。
5. 系统生成新的回放和分析，比较两次执行的人工介入率、返工点和 replay coverage。

关键价值验证：
- 第二次执行所需人工介入更少
- 返工原因更集中、更可解释
- SpecAsset 的价值可以被事件和报告证明

预期产物：
- 两份可对比的回放
- 两份分析报告
- 至少一条 `SpecRevision` 记录
- 一份“调优前后”差异摘要

## Interfaces / Types
场景文档必须能引用：
- `Session`
- `TaskNode`
- `WorkerRun`
- `EventEnvelope`
- `ConflictRecord`
- `ReplayDocument`
- `AnalysisReport`

## Failure Modes
- 如果场景 A 跑不通，说明最小闭环未建立。
- 如果场景 B 无法解释人工介入和重试，说明可观测链条仍然不够完整。

## Observability
- 每个场景都要能列出最少必要事件集合。
- 每个场景都要能标出人工介入点和回放边界。

## Open Questions / ADR Links
- 该文档已纳入“调优前后对比”场景，后续可继续补充更多垂直场景。

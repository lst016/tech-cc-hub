# 32-回放与分析报告规范

## Purpose
定义 `ReplayDocument` 和 `AnalysisReport` 的目标、结构和生成流程。

## Scope
本文件覆盖从事件到时间线、再到 Markdown 产物的转换规则。

## Actors / Owners
- Owner: Analyzer
- Readers: 后端、前端、调优者、复盘读者

## Inputs / Outputs
- Inputs: `EventEnvelope`、Session 状态、Task Graph、ConflictRecord
- Outputs: `ReplayDocument`、`AnalysisReport`

## Core Concepts
- `ReplayDocument`: 面向人类阅读的执行证据。
- `AnalysisReport`: 面向优化决策的结构化结论。
- `Evidence Link`: 报告结论回链到事件和回放片段的机制。

## Behavior / Flow
生成流程：
1. 收集 Session 范围内的标准事件。
2. 构建 `Timeline`，并按 Session / Task / Worker 聚类。
3. 生成 `ReplayDocument`，保留关键行为与人机互动链。
4. 基于时间线和状态统计生成 `AnalysisReport`。
5. 报告中的结论必须能指回事件证据或回放小节。

## Interfaces / Types
`ReplayDocument` 建议结构：
- 概览
- 任务图摘要
- 时间线
- 关键转折点
- 人工介入记录
- 冲突与解决
- 产物索引

`AnalysisReport` 建议结构：
- 基本信息
- 核心指标
- 人工干预分析
- 错误与返工分析
- Workflow / Skills 影响判断
- 改进建议

## Failure Modes
- 如果报告结论不能回链事件，分析将沦为主观总结。
- 如果回放只显示最终摘要，不显示关键转折点，调优价值会大幅下降。

## Observability
- 记录回放和报告生成耗时、失败原因、引用的事件数量和覆盖率。

# Right Context Rail Workbench Design

**Date:** 2026-04-20
**Status:** approved for implementation
**Scope:** 右侧执行分析工作台增强

## 背景

当前右侧栏已经从“指标卡优先”修正为“时间线优先”，但仍然有三个明显缺口：

1. 点击节点后的详情被放在主右栏底部，阅读路径被拉长，用户需要上下滚动才能在“轨迹”和“详情”之间切换。
2. 当前只展示原子节点，缺少“任务级步骤”视角，无法回答“模型打算怎么做”和“后续节点在执行哪个步骤”。
3. 缺少上下文分布视图，用户无法快速判断哪些内容正在吞噬上下文预算，也无法据此优化 prompt、计划或工具链。

这些问题和产品文档中的 Evidence First 原则并不一致。右侧应是执行复盘工作台，而不是一个只能看流水账的附属面板。

## 目标

把右侧区域升级为三层分析工作台：

1. 主右栏显示任务级步骤带与原子执行轨迹。
2. 点击节点后，从右侧再拉出一层详情抽屉，承载完整证据。
3. 通过弹窗查看上下文分布，支持面向优化的审视。

## 设计决策

### 1. 详情改为二级右侧抽屉

- 保留现有 `360px` 右侧主栏作为“总览 + 时间线”。
- 点击任一原子节点后，在主栏左侧展开一个 `320px ~ 360px` 的详情抽屉。
- 详情抽屉展示：
  - 节点标题、状态、轮次、层级
  - 完整 detail 文本
  - 相关 chips
  - 相关任务步骤
  - 关联上下文摘要
- 主栏不再在底部常驻“大块步骤详情”。

原因：这样不会挤压聊天主区，也能维持“先扫时间线，再看细节”的阅读顺序。

### 2. 任务级步骤带

- 当 assistant 文本中存在明确计划列表时，抽取任务级步骤。
- 仅接受显式结构，如：
  - `1. ... 2. ... 3. ...`
  - `Step 1: ...`
  - `第一步 / 第二步 / 第三步`
- 任务步骤位于原子时间线之上，默认横向/纵向紧凑显示。
- 每个步骤展示：
  - 步骤标题
  - 状态：未开始 / 进行中 / 已完成
  - 关联原子节点数
- 原子节点会尝试归属到任务步骤：
  - 优先依据步骤文本关键词
  - 辅以工具类型和节点语义（读/写/验证/汇总）

原因：这能同时回答“模型的计划是什么”和“它实际在哪一步”。

### 3. 上下文分布弹窗

- 在右栏头部提供“上下文分布”入口。
- 弹窗展示会话最近上下文的近似构成：
  - 用户提示
  - AI 计划
  - 思考
  - 工具输入
  - 工具输出
  - 中间结果
  - 最终结果
  - 附件
  - 权限/人工介入
- 每类展示：
  - 近似字符量
  - 占比
  - 条目数
  - 最近样本预览
- 明确标注“这是近似分布，不是模型真实 token 计费拆账”。

原因：用户真正需要的是优化视角，而不是一堆抽象指标。

## 数据模型

在 `activity-rail-model.ts` 中新增三类输出：

1. `taskSteps`
   - 来自 assistant 显式计划
   - 包含状态、步骤文本、关联 timeline ids

2. `selectedItemContext`
   - 给详情抽屉使用
   - 包含节点关联的任务步骤和上下文摘要

3. `contextDistribution`
   - 分类后的上下文近似分布
   - 包含字符量、占比、数量、样本

## 交互

- 点击步骤：
  - 高亮该步骤
  - 优先过滤并聚焦其关联的原子节点
- 点击原子节点：
  - 打开/切换详情抽屉
- 关闭详情抽屉：
  - 返回主时间线浏览
- 打开上下文分布：
  - 弹窗覆盖中间区和右侧区
  - 不销毁当前时间线选中状态

## 验证

本次实现至少覆盖：

1. 纯模型测试
   - 可识别显式任务步骤
   - 能生成上下文分布
   - 能把任务步骤和原子节点建立关联

2. 构建与静态检查
   - `npm run transpile:electron`
   - `npm run build`
   - `eslint`

3. Electron 真窗口验证
   - 主右栏出现任务步骤带
   - 点击原子节点时出现二级详情抽屉
   - 可打开上下文分布弹窗

## 非目标

- 不在这轮引入真正的 Task Graph 持久化对象
- 不做跨 Session 的上下文比较
- 不做真实 token 级账单分摊

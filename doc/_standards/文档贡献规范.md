# 文档贡献规范

## Purpose
定义 CLAW 1.0.0 文档体系的日常维护规则，保证新文档、新决策和已有规范能稳定演进，而不是越写越乱。

## Scope
本规范适用于 `doc` 下所有受管理文档，包括架构、规范、运营、ADR、模板、标准以及版本化产品文档。
本规范不回溯改造历史草稿目录 `doc/`。

## Actors / Owners
- Owner: CLAW Core
- Readers: 所有新增或修改 `doc` 受管理文档的贡献者

## Inputs / Outputs
- Inputs: 新需求、新决策、新规范或对既有文档的修订
- Outputs: 合规的 Markdown 文档、front matter、索引更新和 ADR 记录

## Core Concepts
- `semantic directories`: 语义化目录组织，目录表达用途，编号表达阅读顺序。
- `doc owner`: 每份文档只保留一个主 owner，避免多人重复定义同一概念。
- `source of truth`: 一个概念只能有一份主规范，其他文档只引用、不重写。

## Behavior / Flow
### 目录使用规则

| Directory | Use |
|---|---|
| `00-overview/` | 产品边界、问题定义、术语、索引 |
| `10-architecture/` | Context/Container/Component 级架构图 |
| `20-specs/` | 核心契约、类型、状态机、协议 |
| `30-operations/` | 前端信息架构、流程、回放、路线图、验收 |
| `adr/` | 正式架构决策记录 |
| `_templates/` | 模板，不承载正式结论 |
| `_standards/` | 规范、约定、贡献规则 |
| `_tools/` | 校验或自动化工具 |

### 编号规则

- `00-09`: 总览和产品定义
- `10-19`: 架构图与系统分层
- `20-29`: 协议和类型契约
- `30-39`: 运行、交付、验收与演进
- `ADR-xxx`: 决策记录

新增文档时：
- 优先插入最合理的编号段
- 不为了“看起来连续”重排整仓编号
- 只有当信息结构发生明显变化时才新增新段

### 什么时候新建文档

应新建文档的情况：
- 出现一个新的一级主题或稳定对象 owner
- 新内容会让现有文档明显跨主题膨胀
- 需要单独建立可追踪的规范边界

应修改既有文档的情况：
- 只是细化已有主题
- 只是补字段、补示例、补链接
- 新内容仍属于当前文档的主职责

### 修改文档时必须做的事

1. 更新 front matter 中的 `last_updated`
2. 保持 `title` 与 H1 一致
3. 如果新增的是正式规范，更新 [03-文档索引.md](../00-overview/03-%E6%96%87%E6%A1%A3%E7%B4%A2%E5%BC%95.md)
4. 如果新增的是架构决策，更新 [35-ADR目录.md](../30-operations/35-ADR%E7%9B%AE%E5%BD%95.md)
5. 如果新增概念，检查 [02-术语表.md](../00-overview/02-%E6%9C%AF%E8%AF%AD%E8%A1%A8.md) 是否需要补充

### 版本化产品文档规则

1. 产品与开发文档统一放入 `doc/40-product/<semver>/`
2. `<semver>` 使用 `major.minor.patch` 格式，例如 `1.0.0`
3. 同一版本下继续按 `requirements / quality / epics / delivery / traceability` 拆分，不回退为单一大文档
4. 新增 FR/NFR/Epic/开发文档后，必须同步更新对应索引和追踪矩阵

### 什么时候必须写 ADR

以下情况不允许只改正文而不写 ADR：
- 改变 AgentOS 统一能力模型
- 改变事件模型或回放证据链原则
- 改变 SpecAsset / RuntimeAsset 的治理方式
- 改变存储真相来源
- 改变递归任务图的预算或停止策略

## Interfaces / Types
### Front Matter Checklist

每份文档必须具备：
- `doc_id`
- `title`
- `doc_type`
- `layer`
- `status`
- `version`
- `last_updated`
- `owners`
- `tags`

### 校验命令

```bash
python3 doc/_tools/validate_frontmatter.py
```

## Failure Modes
- 不更新索引，会让新文档变成孤儿页。
- 把新主题硬塞进旧文档，会让文档再次膨胀成“大总文档”。
- 不写 ADR，会让关键决策回到口头状态。

## Observability
- 每次新增或修改文档后都应运行 front matter 校验脚本。
- 后续可以扩展脚本，增加坏链检查和编号检查。

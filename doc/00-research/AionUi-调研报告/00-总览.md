## 一、项目概述

### 1.1 基本信息

| 属性 | 值 |
|------|-----|
| 名称 | AionUi |
| 定位 | Free, local, open-source 24/7 Cowork app with AI Agents |
| 许可证 | Apache-2.0 |
| 技术栈 | Electron 39 + React 19 + UnoCSS + Arco Design |
| 构建工具 | electron-vite |
| 多语言支持 | 中/英/日/韩等 10+ 语言 |

### 1.2 核心特性

- **多 Agent 统一界面**: 支持 Claude Code, Codex, Qwen Code, Kiro, Hermes Agent, Snow CLI, Cursor Agent 等 16+ Agent
- **内置 Agent 引擎**: 无需单独安装 CLI，开箱即用
- **远程访问**: WebUI + Telegram / Lark / DingTalk / WeChat / WeCom
- **计划任务**: Cron 24/7 无人值守运行
- **多进程架构**: Main + Renderer + Worker 分离

### 1.3 与 tech-cc-hub 对比

| 维度 | AionUi | tech-cc-hub |
|------|--------|-------------|
| Agent SDK | 多 Agent 适配器 | Claude Agent SDK |
| UI 框架 | Arco Design + UnoCSS | Radix + Tailwind CSS v4 |
| 多 Agent | 原生支持 | 单 Agent |
| 远程访问 | WebUI + 消息平台 | 无 |
| Cron | 内置 | 无 |
| MCP | 完整实现 | 无 |

## 二、技术架构

### 2.1 多进程模型

```
┌─────────────────────────────────────────────────────┐
│  Main Process (src/process/)                       │
│  - 应用逻辑、数据库、IPC 处理                        │
│  - 无 DOM API                                       │
├─────────────────────────────────────────────────────┤
│  Renderer Process (src/renderer/)                   │
│  - React UI                                         │
│  - 无 Node.js API                                   │
├─────────────────────────────────────────────────────┤
│  Worker Processes (src/process/worker/)             │
│  - Gemini Worker                                    │
│  - Lifecycle Runner                                 │
│  - 后台 AI 任务                                      │
└─────────────────────────────────────────────────────┘
```

**IPC 通信通过 preload bridge**，所有跨进程通信必须走 IPC。

### 2.2 运行模式

AionUi 支持 4 种运行模式：

| 模式 | 说明 |
|------|------|
| `start` / `cli` | Electron 桌面模式 |
| `webui` | Electron 无窗口，纯 WebUI |
| `server` | 纯 Node.js，无 Electron |
| `pet` | 轻量助手模式 |

### 2.3 WebUI 服务器

位置: `src/process/webserver/`

- Express + WebSocket 实时通信
- JWT 认证
- 支持远程浏览器访问 Agent UI
- 10 个 Electron-only bridge 在纯 server 模式下不可用

### 2.4 IPC 桥接架构

**核心文件**: `src/common/adapter/main.ts`

```typescript
// 50MB 最大 Payload 保护
const MAX_IPC_PAYLOAD_SIZE = 50 * 1024 * 1024;

bridge.adapter({
  emit(name, data) {
    // 1. 发送给所有 Electron BrowserWindow
    // 2. 通过 WebSocket 广播给远程客户端
  },
  on(emitter) {
    // 保存 emitter 引用供 WebSocket 处理
  },
});
```

**桥接事件数据结构**:
```typescript
interface BridgeEventData {
  name: string;
  data: unknown;
}
```

## 四、代码规范

### 4.1 目录结构约束

> 单个目录最多 10 个直接子项，超过需按职责拆分

### 4.2 命名约定

| 类型 | 约定 | 示例 |
|------|------|------|
| 组件 | PascalCase | `Button.tsx` |
| 工具函数 | camelCase | `formatDate.ts` |
| Hooks | camelCase + use 前缀 | `useTheme.ts` |
| 常量文件 | camelCase | `constants.ts` |
| 类型文件 | camelCase | `types.ts` |

### 4.3 UI 规范

- **组件库**: `@arco-design/web-react`（禁止原始 HTML 交互元素）
- **图标**: `@icon-park/react`
- **样式**: UnoCSS 原子类 + CSS Modules
- **颜色**: 必须使用 `uno.config.ts` 的语义化 token

### 4.4 TypeScript 规范

- 严格模式（无 `any`，无隐式返回）
- 路径别名：`@/*`, `@process/*`, `@renderer/*`
- 推荐 `type` 而非 `interface`

### 4.5 测试规范

- 框架: Vitest 4
- 运行: `bun run test`
- 覆盖率目标: ≥ 80%

## 六、关键差异与可借鉴点

### 6.1 可直接借鉴

| 功能 | AionUi 实现 | 移植建议 |
|------|-------------|----------|
| **多 Agent 支持** | AgentRegistry + 适配器模式 | 为 tech-cc-hub 设计 Agent 抽象层 |
| **协议转换器** | OpenAI ↔ Anthropic ↔ Gemini | 统一 API 调用接口 |
| **Cron 调度** | croner + CronBusyGuard | 计划任务支持 |
| **MCP 协议** | 完整 MCP 服务实现 | 直接参考或复用 |
| **WebUI 远程** | Express + WebSocket + JWT | 远程协作能力 |
| **IPC Payload 保护** | 50MB 限制 + 错误通知 | 防止主进程阻塞 |

### 6.2 架构差异（不可直接移植）

| 差异点 | 说明 |
|--------|------|
| UI 框架 | Arco Design vs Radix，组件体系完全不同 |
| 样式方案 | UnoCSS vs Tailwind CSS |
| 进程模型 | 额外的 lifecycleRunner worker |
| 多语言 | 完整 i18n skill 系统 |

### 6.3 技术亮点

1. **IPC 序列化保护**: 超大 Payload 直接丢弃并通知，而非阻塞
2. **多实例端口自增**: Vite 端口冲突时自动递增
3. **Sentry 源码映射**: 构建时自动规范化路径
4. **MCP 服务器独立构建**: 主进程打包后单独构建 MCP 服务
5. **PR 自动化**: bot label 驱动的 PR 状态机

## 八、总结与建议

### 8.1 高优先级借鉴

1. **MCP 服务实现** — 我们完全缺失，MCP 能力对 Agent 生态至关重要
2. **Cron 调度系统** — 24/7 无人值守是强需求
3. **API 适配层模式** — 统一的多后端支持

### 8.2 中优先级借鉴

4. **协议转换器** — 未来可能需要 Gemini/Qwen 支持
5. **IPC 保护机制** — 防止超大 Payload 阻塞
6. **多 Agent 架构** — 未来扩展预留

### 8.3 低优先级（架构差异大）

7. **Arco Design UI** — 与我们的 Radix 体系不兼容
8. **UnoCSS 样式** — 重写成本高
9. **WebUI 远程访问** — 复杂度高，收益待评估

### 8.4 下一步行动

1. **Phase 1**: 详细分析 `mcpServices/` 实现细节
2. **Phase 2**: 设计 tech-cc-hub MCP 集成方案
3. **Phase 3**: Cron 系统设计文档

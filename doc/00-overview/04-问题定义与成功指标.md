# 04-问题定义与成功指标

## Purpose
把 CLAW 的产品问题、目标用户、替代方案、Jobs-to-be-Done 和成功指标写清楚，确保后续架构和路线图服务的是明确业务价值，而不是抽象技术能力。

## Scope
本文件定义 v1 的问题空间、目标用户分层、核心价值假设和 3/6/12 个月成功标准。
本文件不定义协议、字段 schema 或实现细节。

## Actors / Owners
- Owner: Product + Architecture
- Primary Readers: 产品、架构、项目负责人
- Secondary Readers: 前端、后端、分析与调优实现者

## Inputs / Outputs
- Inputs: QCon 学习结论、已有架构文档、单人研发者的真实使用痛点
- Outputs: 问题定义、目标用户、JTBD、替代方案、核心指标、优先级依据

## Core Concepts
- `Primary User`: 高频使用 Claude Code / Codex 的单人研发者，已经能把 Agent 用起来，但还没有稳定的方法让 Agent 行为长期可控、可复盘、可持续调优。
- `Secondary User`: 对 Agent 工作流有治理需求的技术负责人或平台建设者，但不属于 v1 的主目标用户。
- `Current Alternatives`: 直接使用 Claude Code / Codex、分散的 prompt 模板、零散 markdown 记录、个人经验法则、临时脚本和 Git 提交记录。
- `Core Product Thesis`: 用户真正缺的不是“另一个 Agent”，而是一个能把 Agent 使用过程变成可回放、可追责、可调优系统的软件层。

## Behavior / Flow
### Problem Definition

当前高频 Agent 使用者会反复遇到 4 类问题：

1. `执行不透明`
   任务复杂后，用户很难解释 Agent 为什么走到当前状态，也很难复盘失败拐点。
2. `调优不可沉淀`
   用户会积累 prompt、skills、规则和经验，但它们是散的，难以版本化、复用和持续优化。
3. `多任务协作不可控`
   当任务开始拆分、并行或跨 AgentOS 时，用户很难看清依赖、冲突、返工和人工介入链路。
4. `结果无法形成证据闭环`
   最终只看见“成功/失败”，却拿不到足够强的时间线、事件和分析材料来持续改进。

### Why Now

- AgentOS 已经足够可用，用户痛点从“能不能用”转向“能不能稳定地用好”。
- QCon 中关于 AI Coding、Spec-driven、上下文工程、可观测和企业可控性的讨论，说明市场心智正在形成。
- 对单人研发者来说，现在最大的缺口不是底层模型，而是上层工作方法和产品化治理。

### Jobs-to-be-Done

#### JTBD 1: 复杂任务执行可解释
当我把复杂开发任务交给 Agent 时，我希望看到任务是如何被拆分、执行、失败、重试和合并的，这样我才能信任结果并在必要时及时接管。

#### JTBD 2: 调优经验可复用
当我发现某种 workflow、skill 或 policy 能明显提升结果时，我希望把它固化成可版本化资产，并在下一次相似任务中复用和验证。

#### JTBD 3: 多 Agent 协作可治理
当同一个目标由多个 AgentOS 或多个子任务共同推进时，我希望看到明确的任务图、冲突记录和人工决策链，而不是把所有复杂性压回聊天窗口。

### Target User Tiers

| Tier | User | Need | Priority |
|---|---|---|---|
| `P1` | 高频单人研发者 | 可观测、可回放、可调优 | Highest |
| `P2` | 技术负责人 / 平台建设者 | 规范化 workflow、策略治理、后续团队扩展 | Later |
| `P3` | 普通轻度 Agent 用户 | 简单聊天增强 | Out of v1 focus |

### Alternatives and Gaps

| Current Alternative | What It Solves | Where It Fails |
|---|---|---|
| 直接用 Claude Code / Codex | 基础执行能力 | 缺少统一回放、跨任务治理、SpecAsset 沉淀 |
| 零散 prompt / markdown | 个人经验积累 | 无法结构化复用、无证据链 |
| Git 提交和终端历史 | 部分结果留痕 | 缺乏任务级和决策级上下文 |
| 手工记录复盘 | 事后总结 | 成本高、遗漏多、难持续 |

## Interfaces / Types
### v1 Core Value Hypotheses

| Hypothesis ID | Statement |
|---|---|
| `H1` | 如果用户能看到结构化时间线和回放，复杂任务的信任度和接管效率会显著提升。 |
| `H2` | 如果 workflow / skills / policies 可版本化，用户会开始形成可复用的方法资产，而不是一次性 prompt。 |
| `H3` | 如果跨 AgentOS 的任务和冲突可以被统一呈现，用户愿意把更复杂的任务交给系统。 |

### Success Metrics

#### 3 Months
- 至少能稳定支持 `单 Session -> 事件流 -> 回放文档` 闭环。
- 至少形成 3 类可复用 `SpecAsset`。
- 复杂任务回放生成成功率达到 `80%+`。

#### 6 Months
- 至少支持 2 个 AgentOS 的统一能力视图。
- 相似任务二次执行时，人工介入次数较首次下降 `20%+`。
- 至少 `50%` 的复杂任务使用了版本化 `SpecAsset`。

#### 12 Months
- 用户能稳定比较“调优前/调优后”效果差异。
- 复杂任务的 replay coverage 保持在 `90%+`。
- 至少形成一套可迁移到团队场景的 workflow / policy 基线。

#### Product-Level Leading Indicators
- `replay_generation_rate`
- `replay_coverage_rate`
- `spec_asset_reuse_rate`
- `human_intervention_rate`
- `rework_rate`
- `cross_agent_task_success_rate`

## Failure Modes
- 若只讨论系统能力而不定义目标用户和痛点，CLAW 会变成无边界的平台愿景。
- 若没有成功指标，路线图会只能按“功能做完没”推进，而不是按“价值成立没”推进。
- 若把 v1 用户扩到所有开发者，会稀释最有价值的高频用户场景。

## Observability
- 所有产品指标都应能回链到 `EventEnvelope`、`ReplayDocument` 或 `SpecRevision`。
- 不允许只用主观反馈定义“好不好用”。

# 01-设计原则与非目标

## Purpose
固定 CLAW 的架构红线，避免后续规范在“做平台”还是“做壳子”之间反复摇摆。

## Scope
本文件定义必须长期坚持的设计原则与明确不做的事情。

## Actors / Owners
- Owner: Architecture
- Readers: 所有参与写 spec 或实现 CLAW 的成员

## Inputs / Outputs
- Inputs: 产品定位、用户偏好、现有草稿
- Outputs: 架构红线、非目标、优先级顺序

## Core Concepts
- `上层软件层`: 不接管 LLM 推理或底层工具系统，只增强控制面和观测面。
- `本地优先`: 本机文件系统是 v1 的事实真相。
- `双中心`: SpecAsset 和 RuntimeAsset 并列为一等资产。
- `统一优先`: 优先抽象 Claude Code 和 Codex 的通用能力，再保留扩展位。
- `回放优先`: 先保证事件到回放闭环，再做高阶分析。

## Behavior / Flow
后续所有规范都必须满足以下原则：

1. CLAW 是 AgentOS 的半托管控制层，不是自研底层执行内核。
2. v1 是 `local-first / single-user / desktop-first`。
3. SpecAsset 和 RuntimeAsset 必须分层管理、单独建模。
4. 递归任务图能力允许持续细拆，但必须经过策略层约束，而不是无边界失控。
5. 同一概念只能有一个主规范 owner，避免多处重复定义。

## Interfaces / Types
本文件约束下列规范的方向：
- `20-AgentOS集成规范`
- `21-统一能力模型`
- `22-任务图与递归拆分规范`
- `24-事件模型与可观测规范`

## Failure Modes
- 非目标 1: 不实现新的通用推理 runtime。
- 非目标 2: 不在 v1 引入多租户、团队权限和云真源。
- 非目标 3: 不用一个超长文档混写产品、架构、协议和实现细节。
- 非目标 4: 不强迫所有 AgentOS 在底层执行细节上完全同构。

## Observability
- 每份后续规范都要声明自己会产生或消费哪些事件。
- 每个用户看见的结论都应能追溯到事件、状态或 SpecAsset 版本。

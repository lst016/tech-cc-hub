# 执行链路性能优化 Spec（2026-07-08）

更新时间：2026-07-08

关联会话反馈：`session 18bb89b9-e7b2-45fb-8789-78cb14d2651e`（用户反馈“使用很慢”）

## 关键决策（2026-07-08 用户裁定）

1. **保留 `ActivityRail` shell**，不砍掉。它现在的角色是"多 tab workspace"（usage / preview / git / terminal / workflow-agent），不是"消息流右栏"——消息渲染已迁到中心主区（`App.tsx:1814-1977` 的 `renderEntries` + `MessageCard` / `ProcessGroupCard` / `WorkflowAgentCard`）。
2. **按 tab 按需加载**：用户没点开的 tab，**不构建其依赖的派生数据、不渲染其内容**；用户点开了，才跑构建。用户原话："那几个 tab 如果没打开我希望就不要使用，打开了那就用，这样不用浪费性能了。"
3. **消息流右栏渲染**早已移除（commit `bffc1c3 perf: 移除 600 条消息上限`），中心列才是热路径；本 spec 的优化重心在主区 + IPC + store，右栏只做"按需"。

## 背景

收到真实用户反馈：长会话、多工具调用场景下，技术栈 hub 的执行链路体感偏慢，TTFT 后流式卡顿、中心聊天列刷新存在肉眼可见的延迟。

排查后定位根因：

- 一次 Claude SDK 模型回合会产生 100+ 条 `stream_event`（含 token delta / thinking / tool_use / tool_result），当前实现里 **每条事件都走完整的主进程 → IPC → 渲染端 → Store → React 重渲染链路**，没有在任何一个节点做合并或节流。
- `ActivityRail` 即使当前选中 tab 不是 usage（usage 才有重计算需求），仍然在 useMemo 里跑 `buildActivityRailModel` + `visibleTimeline` + `analysisCards` + `materialStatusItems`，属于浪费。
- `appendMessagesToSession` 走全量 spread，大 session 下退化明显。

## 目标

1. 把单回合 IPC 次数从 100+ 降到 10–20 次（按事件类型分桶合并）。
2. 把渲染端 store 更新从逐事件改为时间片合并（≥30fps 上限）。
3. 消除 SQLite 主线程写入抖动（持久化与广播解耦）。
4. 减少大 session 下中心列 React 重渲染面积（`appendMessagesToSession` 不再 O(N) spread，`MessageCard` 自定义 areEqual）。
5. 给用户消息加乐观 UI，省掉发送→主进程回环的 100–300ms 等待。
6. **ActivityRail tab 按需加载**：未打开的 tab 不构建其派生数据（核心是 `buildActivityRailModel`），打开的 tab 才构建。

## 非目标

- **不砍掉 `ActivityRail` shell**，保留作为多 tab workspace 的容器。
- 不重做 ActivityRail UI 结构、不拆 EventCard 大组件（属于打磨轮）。
- 不把 markdown 渲染挪到 Web Worker（成本/收益不匹配，本轮不做）。
- 不改 IPC 协议（保持 `server-event` 字符串 + JSON 通道）。
- 不动 Claude Agent SDK 的 `query()` 调用本身，目标是 hub 的“壳”而不是模型调用。
- 不把 `buildActivityRailModel` 拆成 timeline/executionSteps/analysisCards 三段 memo（被 tab-gating 取代）。

每条 stream event（包含一个 token delta）当前都走"上半段"完整链路；右栏若被打开且当前 tab 是 usage，则多走一段 `buildActivityRailModel` 全量遍历。优化重点是上半段 + 右栏 gating。

## 现状瓶颈（按调用点）

### B1. 主进程 `broadcast()` 完全无批处理
`src/electron/ipc-handlers.ts:197`

```ts
function broadcast(event: ServerEvent) {
  const payload = JSON.stringify(event);
  for (const win of BrowserWindow.getAllWindows()) {
    win.webContents.send("server-event", payload);
  }
}
```

一次模型响应 100+ `stream_event` 就是 100+ 次 `JSON.stringify + IPC roundtrip + JSON.parse`。

### B2. SQLite 写入逐事件
`src/electron/libs/session-store.ts:472 recordMessage`

transient 事件（`stream_event` / `thinking_tokens`）已经被 `isTransientStreamEventMessage` 跳过，但 `assistant` / `user` 类型仍然单条 `INSERT OR IGNORE`，每个工具调用都触发一次 fsync。

### B3. workflow.run.updated 逐事件广播
`src/electron/ipc-handlers.ts:694 trackWorkflowRunsFromStreamMessage` → 每个 stream.message 都跑 `upsertWorkflowRun + broadcast`。

### B4. 渲染端 `appendMessagesToSession` 数组 spread
`src/ui/store/useAppStore.ts:372`

```ts
const messages = [...session.messages, ...nextMessages];
```

600 条消息的 session 每个 batch 拷贝 600 个引用。

### B5. ActivityRail 不分 tab 全量跑 `buildActivityRailModel`
`src/ui/components/ActivityRail.tsx:1329-1332`

```ts
const model = useMemo(
  () => buildActivityRailModel(modelSession, modelSession?.permissionRequests ?? [], ""),
  [modelSession],
);
```

`buildActivityRailModel` 是 session 维度的全量遍历（详见 `src/shared/activity-rail-model.ts:1636`），但当前实现**不管用户当前选中的是哪个 tab 都跑**。`preview / git / terminal / workflow-agent:*` 这几个 tab 用不到 `model`（它们各自有自己的 panel），却照样吃 CPU + 触发下游 `visibleTimeline / analysisCards / activeAgentNodes / materialStatusItems / attachmentSummary` 全部重算，再被 React 丢掉。

### B6. ActivityRail DetailDrawer / ContextDistributionModal 跨 tab 残留

`src/ui/components/ActivityRail.tsx:1410-1431`

`selectedTimelineId` / `showContextModal` 是 ActivityRail 内部 state，切到非 usage tab 不会自动清掉，导致切回 usage 时仍渲染 stale 数据；进一步加重 B5 的浪费（抽屉一开就必须有 model，model 不能 gapped）。

### B7. MessageCard memo 粒度太粗
`src/ui/components/EventCard.tsx:2321`

```ts
export const MessageCard = memo(MessageCardBase);
```

`partialMessage` 变化时所有卡片都收到新 props，浅比较失败 → 全量卡片重渲染。

### B8. 用户消息等待回环
当前用户点发送 → 主进程处理 → 返回 `stream.user_prompt` → 才显示用户消息。这条链路能省 100–300ms。

### B9. 主进程热路径上的杂活
- `ipc-handlers.ts:594 trackFigmaAuthToolState` 每个 event 都跑
- `runner.ts:902 console.info` 在 `canUseTool` 内每次工具调用都打
- `ipc-handlers.ts:579 scheduleCodeGraphAutoSyncAfterTurn` 在每个 turn 边界跑

## 优化方案（按 ROI 排序）

### P0 — 立即做，立竿见影

#### 方案 A：主进程 IPC 批处理（最大收益点）

**目标：** 把单回合 IPC 次数从 100+ 降到 10–20。

**位置：** `src/electron/ipc-handlers.ts`

**做法：**

```ts
const pendingBroadcastByType = new Map<ServerEvent["type"], ServerEvent[]>();
const BROADCAST_FLUSH_MS = 50;
let broadcastTimer: NodeJS.Timeout | null = null;

function broadcast(event: ServerEvent) {
  // transient 流式事件：按 type 分桶，50ms 合并
  if (isTransientStreamEvent(event)) {
    const bucket = pendingBroadcastByType.get(event.type) ?? [];
    bucket.push(event);
    pendingBroadcastByType.set(event.type, bucket);
    if (!broadcastTimer) {
      broadcastTimer = setTimeout(flushBroadcast, BROADCAST_FLUSH_MS);
    }
    return;
  }
  // 关键事件：立即发，不批处理
  sendImmediate(event);
}

function flushBroadcast() {
  broadcastTimer = null;
  for (const [type, events] of pendingBroadcastByType) {
    sendImmediate({ type, payload: events } as ServerEvent);
  }
  pendingBroadcastByType.clear();
}
```

**token delta 类**更进一步：只保留最后一个 delta，不发中间过程。渲染端只要最新文本。

**预期收益：** IPC 开销 -50%。

#### 方案 L：ActivityRail tab 按需加载（用户裁定，本轮必做）

**目标：** ActivityRail 不分 tab 跑 `buildActivityRailModel` 及其派生数据，是浪费。改为：只有当前选中 tab 是 `usage`（或 `showContextModal === true`）才构建 model，其他 tab 直接 `model = null`，相关派生全部 short-circuit。

**位置：** `src/ui/components/ActivityRail.tsx:1322-1406`

**做法：**

```ts
// 1. 引入 tab 判定
const isUsageTab = selectedTab === "usage";
const needsModel = isUsageTab || showContextModal;

// 2. gate model 构建（line 1329-1332）
const model = useMemo(
  () => (needsModel && modelSession)
    ? buildActivityRailModel(modelSession, modelSession?.permissionRequests ?? [], "")
    : null,
  [needsModel, modelSession],
);

// 3. 派生全部 null-guard
const visibleTimeline = useMemo(
  () => model ? model.timeline.slice(0, INLINE_TRACE_TIMELINE_LIMIT) : [],
  [model],
);

const selectedItem =
  (isUsageTab && model && selectedTimelineId)
    ? model.timeline.find((item) => item.id === selectedTimelineId) ?? null
    : null;
const relatedSteps =
  (isUsageTab && model && selectedItem)
    ? model.executionSteps.filter((step) => step.timelineIds.includes(selectedItem.id))
    : [];

const analysisCards = useMemo(() => {
  if (!model) return [];
  if (!globalError) return model.analysisCards;
  return [{ id: "global-error", title: "界面异常", tone: "error" as const, detail: globalError }, ...model.analysisCards];
}, [globalError, model]);

const activeAgentNodes = useMemo(() => {
  if (session?.status !== "running" || !model) return [];
  const seen = new Set<string>();
  return model.timeline.filter((item) => {
    if (item.nodeKind !== "agent_progress" || item.metrics.status !== "running") return false;
    const key = item.parentTaskId || item.agentDescription || item.title;
    if (seen.has(key)) return false;
    seen.add(key);
    return true;
  });
}, [model?.timeline, session?.status]);

const attachmentSummary = model
  ? summarizeAttachments(model.contextSnapshot.latestAttachments.map((a) => a.name))
  : null;

const materialStatusItems = useMemo(
  () => model ? buildMaterialStatusItems(model, partialMessage) : [],
  [model, partialMessage],
);

// 4. 切走 usage 时清掉 drawer / modal state（避免 stale）
useEffect(() => {
  if (!isUsageTab) {
    setSelectedTimelineId(null);
    setShowContextModal(false);
  }
}, [isUsageTab]);

// 5. drawer / modal 渲染限定在 usage tab
{showContextModal && model && (
  <ContextDistributionModal
    title={model.contextModalTitle}
    totalChars={model.contextDistribution.totalChars}
    buckets={model.contextDistribution.buckets}
    onJumpToNode={(timelineId) => setSelectedTimelineId(timelineId)}
    onClose={() => setShowContextModal(false)}
  />
)}
{isUsageTab && selectedItem && model && (
  <DetailDrawer
    title={model.detailDrawerTitle}
    item={selectedItem}
    relatedSteps={relatedSteps}
    latestPrompt={model.contextSnapshot.latestPrompt}
    attachmentSummary={attachmentSummary}
    partialMessage={partialMessage}
    onClose={() => setSelectedTimelineId(null)}
  />
)}
```

**前置状态：** preview / git / terminal / workflow-agent 各 panel 本身已经走 `lazy + Suspense`（`ActivityRail.tsx:1466-1514`），本方案不需要再加一层动态 import。

**预期收益：** 当用户把右栏切到非 usage tab 时，`buildActivityRailModel` + 5 个下游派生 useMemo 全部 short-circuit；对 preview/git/terminal/workflow-agent 这些重 panel tab，渲染期间主线程 CPU 占用预计下降 30–50%。

**验收：**

- 在 ActivityRail 切到 `preview` 后，`buildActivityRailModel` 在 React Profiler 中不再出现；
- `useDeferredValue + useMemo` 在切到非 usage tab 后不再触发 `buildActivityRailModel`；
- 切回 `usage` 立即重建 model，不应有可见卡顿；
- `selectedTimelineId` / `showContextModal` 切走 usage 后自动清零，切回时为干净状态。

#### 方案 C：`appendMessagesToSession` 改 push/immer

**目标：** 消除 O(N) 数组拷贝。

**位置：** `src/ui/store/useAppStore.ts:363`

**做法：** 引入 `immer` middleware 或直接：

```ts
function appendMessagesToSession(session: SessionView, nextMessages: StreamMessage[]): SessionView {
  return {
    ...session,
    messages: session.messages.concat(nextMessages),  // concat 比 spread 略快
    latestPlan: deriveLatestPlanSnapshot(session.id, nextMessages, session.latestPlan),
  };
}
```

或上 immer middleware 后用 `state.sessions[id].messages.push(...next)`。React 只关心引用变化。

**预期收益：** 大 session 渲染 -30%。

#### 方案 B：SQLite 写入合并

**目标：** 消除 fsync 抖动。

**位置：** `src/electron/libs/session-store.ts:472 recordMessage`

**做法：** 加 `pendingMessagesBySession: Map<sessionId, StreamMessage[]>` + 100ms flush，用 `db.transaction()` 包：

```ts
private flushPendingWrites() {
  const transaction = this.db.transaction(() => {
    for (const [sessionId, messages] of this.pendingMessagesBySession) {
      for (const msg of messages) {
        this.persistMessage(sessionId, msg);
      }
    }
  });
  transaction();
  this.pendingMessagesBySession.clear();
}
```

**预期收益：** 主线程 DB 抖动消除。

### P1 — 一两天工作量，中等收益

#### 方案 D：workflow.run 去抖

**位置：** `src/electron/ipc-handlers.ts:694`

**做法：** `Map<sessionId+taskId, WorkflowRunPatch>` last-write-wins，50ms flush 一次 `upsertWorkflowRun + broadcast`。

#### 方案 F：乐观 UI

**位置：** `src/ui/store/useAppStore.ts:412 setPrompt` 链路上

**做法：** 用户点发送 → 立即 push 一条临时 `user_prompt` 消息到 `sessions[activeId].messages`（带临时 `historyId`）→ 渲染端立即显示 → 主进程返回时按 `historyId` 替换。

**预期收益：** 用户感知 -100~300ms。

#### 方案 E：MessageCard 自定义 areEqual

**位置：** `src/ui/components/EventCard.tsx:2321`

**做法：**

```ts
const areMessageCardsEqual = (prev: MessageCardProps, next: MessageCardProps) => {
  if (prev.message !== next.message) return false;
  if (prev.partialMessage === next.partialMessage) return true;
  // 只有包含本卡 uuid 的 partialMessage 变化才算变化
  return !next.partialMessage.startsWith(prev.message.uuid ?? "");
};

export const MessageCard = memo(MessageCardBase, areMessageCardsEqual);
```

**预期收益：** 大 session 渲染 -30%。

#### 方案 G：清理主进程热路径

- `ipc-handlers.ts:594 trackFigmaAuthToolState` → debounce 到 turn 结束
- `runner.ts:902 console.info` → release profile 关掉
- `ipc-handlers.ts:579 scheduleCodeGraphAutoSyncAfterTurn` → debounce + 防抖

### P2 — 长期打磨

#### 方案 H：markdown 流式渲染

流式期间先显示原始文本（plain），流结束（出现 `result` 事件）再切到 markdown 渲染。避免每个 token delta 都触发 markdown 增量 parse。

> 原"方案 I：ActivityRail 增量 model"已被方案 L（tab 按需加载）取代，删除。

#### 方案 J：message schema 预归一化

主进程 `recordMessage` 出口把 message 归一化成渲染端直读 schema，渲染端不再做 `isRecord / Array.isArray / 类型断言`。

#### 方案 K：二进制 IPC

改 `MessagePack` 或 `structured clone`，省 30–50% 序列化时间。前提是 IPC 协议层接受破坏性变更。

## 验收与基线

### 实施前先建基线

临时加诊断代码：

```ts
// ipc-handlers.ts broadcast() 入口
console.time(`[ipc] ${event.type}`);

// useAppStore.ts appendMessagesToSession 入口
performance.mark(`batch-${Date.now()}`);
```

跑 `npm run qa:continue` 回归，收集：

1. **单回合 IPC 总数**（基线：约 = event 数；目标：event 数 / 5–10）
2. **单回合 store 更新次数**（基线：约 = batch 数；目标：持平或更低）
3. **主进程 CPU 占用**（开 V8 profiler 抓 hot function）
4. **TTFT**（首 token 时间，毫秒）
5. **流式持续期间的帧率**（DevTools Performance tab 抓 60fps 维持率）

### 每个 P0 改动验收标准

- **方案 L（tab 按需加载）**：
  - ActivityRail 切到 `preview / git / terminal / workflow-agent:*` 后，React Profiler 中 `buildActivityRailModel` 不再出现（火焰图验证）；
  - 切到 `usage` 时 model 立即重建，**无可见卡顿**（首次切换 < 50ms 可接受）；
  - `selectedTimelineId` / `showContextModal` 在离开 usage 时被清空，回归到 usage 不残留；
  - 单元测试覆盖：`isUsageTab=false` 时所有派生为 `null/[]`，render 不抛错。
- **方案 A**：单回合 IPC 数降到 ≤ 20；token delta 路径不再触发 SQLite。
- **方案 B**：单回合 SQLite write 数从 N 降到 ≤ 5；崩溃恢复仍能拿到最后 100ms 内的关键消息。
- **方案 C**：长 session（> 500 条消息）下 `appendMessagesToSession` 单次耗时 < 1ms（火焰图验证）。

### 整体验收

- 用 `session 18bb89b9-...` 这条长会话重放，TTFT 不退化。
- 多会话窗口并发场景（同时跑 3 个 session）主进程 CPU < 60%。
- 中心聊天列在 1 秒 50+ 事件的注入下，渲染端帧率不掉到 30fps 以下。
- **ActivityRail 切换 tab**：从 usage 切到 preview/git/terminal/workflow-agent 之后，主线程 render 期间 `buildActivityRailModel` 不被调用（DevTools Performance 录制验证）。

## 风险与回滚

| 风险 | 触发条件 | 回滚方式 |
|---|---|---|
| IPC 批处理丢事件 | flush 之前 main crash | 加 `isDev()` 路径立即发；release 默认开 |
| SQLite flush 丢消息 | flush 之前 main crash | 保留原始 `recordMessage` 直写通道作为可选 fallback |
| 乐观 UI 重复 | 网络重试 | 按 `historyId` 去重，重复 historyId 不再 push |
| MessageCard areEqual 漏判 | 内容真正变化但 props 浅相等 | 加一组回归 props 矩阵测试 |
| **方案 L tab gating 漏判** | useEffect 时序导致短暂旧 model 引用 | 加防御性 `if (!model) return null`；首次切回 usage 时强制 re-mount |
| **方案 L 切走清状态破坏 UX** | 用户在 usage tab 打开 drawer 后切到 preview 再切回，drawer 不见了 | 评估是否需要"sticky drawer"：切回 usage 时恢复 `selectedTimelineId`（本轮不做） |

## 实施顺序建议

4. **方案 B**（依赖 A 的批处理思想，避免重复 setTimeout）

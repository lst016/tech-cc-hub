# 60-开发流操作沉淀规范

## Purpose
把开发和 QA 过程中出现的“操作偏航”沉淀成可复用规则，避免同类错误重复发生。

## Scope
本文件关注操作方法、验证顺序、证据采集和经验沉淀，不替代功能 PRD、架构设计或实现规范。

## Behavior / Flow
### 适用场景

- 同一功能存在多个可观测表面，但其中只有一个是真实验收面。
- 某个调试路径方便但会导致错误结论。
- 自动化工具能跑通，但很容易作用到错误窗口、错误页面或错误会话。
- 某次偏航已经被识别，希望把它升级成团队固定方法。

### 操作沉淀的标准结构

每次沉淀都至少记录以下内容：

1. `场景`
   - 在什么功能或测试中出现。
2. `错误路径`
   - 当时走错了哪条操作路径。
3. `错误信号`
   - 如何判断这条路径不可信。
4. `正确路径`
   - 后续必须采用的标准方法。
5. `标准命令 / 脚本`
   - 可直接复用的命令或工具入口。
6. `证据要求`
   - 截图、窗口 ID、日志、session id、命令输出。
7. `抽取出的规则`
   - 以后在什么条件下直接套用这条经验。

### 标准沉淀流程

1. 先定义 `真实验收面`
   - Web、Electron、原生客户端、CLI 或 API。
2. 再定义 `精确定位方式`
   - 页面 URL、窗口 ID、session id、进程名、数据库记录。
3. 记录 `错误路径`
   - 不是为了追责，而是为了将来自动绕开。
4. 固化 `正确命令`
   - 优先提供脚本，而不是口头说明。
5. 记录 `证据样式`
   - 例如“必须包含 window id 和截图路径”。
6. 将经验挂回开发文档和操作索引。

### 当前已沉淀案例

#### Electron 客户端验证

- 错误路径：
  - 直接打开 `localhost` 页面验证 Electron 功能。
- 错误信号：
  - 页面空白、浏览器扩展干扰、结果与客户端真实表现不一致。
- 正确路径：
  - 以 `Electron` 真窗口为准，并优先按 `window id` 定位和截图。
- 标准入口：
  - [electron-client-qa-runbook.md](./electron-client-qa-runbook.md)
  - 复盘记录：[Electron客户端验收面偏航.md](../90-archive/postmortems/electron-qa-correction-retro.md)

## Interfaces / Types
- 操作沉淀文档
- QA 复盘模板
- 可复用脚本
- 标准 npm 命令

## Failure Modes
- 如果只在聊天里口头约定，不写成文档和命令，下一轮很容易再次走偏。
- 如果只记录“正确方法”，不记录“错误信号”，后续仍然难以及时识别偏航。
- 如果没有证据格式要求，就会出现截图和日志无法回放的问题。

## Observability
- 每条操作沉淀至少应能回链到：
  - 一个文档入口
  - 一个脚本或命令
  - 一份真实证据

## Open Questions / ADR Links
- 若后续这类沉淀增多，可以独立出 `runbook / incidents` 文档区。

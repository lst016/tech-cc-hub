# 01-发布目标与范围

## Purpose
定义 CLAW `1.0.0` 的范围边界、MoSCoW 优先级和明确的发布非目标。

## Scope
本文件只回答 `1.0.0 做什么 / 不做什么 / 先做什么`。

## Actors / Owners
- Owner: Product
- Readers: 产品、架构、实现负责人

## Inputs / Outputs
- Inputs: 版本目标、路线图切片、产品边界
- Outputs: In Scope、Out of Scope、优先级基线

## Core Concepts
- `Must`: 没有它版本不成立。
- `Should`: 强烈建议进入 1.0.0，但可在资源不足时延后。
- `Could`: 有价值，但不阻塞首版成立。
- `Won't`: 明确不进入 1.0.0。

## Behavior / Flow
### Must Have

| Area | Capability |
|---|---|
| Chat | Claude Code / Codex 二选一，默认 Claude Code |
| Session | 创建、续写、停止、恢复聊天会话 |
| Task Graph | 从聊天 turn 自动生成治理节点，并支持依赖、重试、回写 |
| AgentOS | 至少接通 2 个 AgentOS 的统一视图 |
| Replay | 从事件流生成时间线与回放文档 |
| Analysis | 生成基础分析报告和核心指标 |
| SpecAsset | workflow / skills / policies 可附着到 Session 或 Task |
| Governance | 权限请求、人工介入、冲突记录闭环 |

### Should Have

| Area | Capability |
|---|---|
| Task Graph | 递归拆分预算控制 |
| SpecAsset | 版本比较与修订说明 |
| Replay | 运行对比视图 |
| Workspace | 文件变化和产物回链 |

### Could Have

| Area | Capability |
|---|---|
| Analysis | 调优建议生成 |
| Governance | 规则化自动批准策略 |
| Workspace | Git 同步面板 |

### Won't Have

| Area | Capability |
|---|---|
| Cloud | 多人协作与云真源 |
| Runtime | 自研底层 Agent 内核 |
| Marketplace | 对外插件市场 |
| Mobile | 移动端客户端 |

## Interfaces / Types
### Release Boundaries

- `In Scope`
  - Desktop-first GUI
  - Local-first runtime assets
  - Versioned spec assets
  - Evidence-first replay loop
- `Out of Scope`
  - Team tenancy
  - Cloud-hosted execution
  - Enterprise admin center
  - Billing / monetization

## Failure Modes
- 如果 `Must` 和 `Should` 不分，1.0.0 会自然膨胀。
- 如果不写 `Won't`, 任何上层治理需求都会被误认为本版本交付。

## Observability
- 发布范围是否被破坏，应通过需求追踪矩阵和迭代计划中的 `out-of-scope drift` 检查。

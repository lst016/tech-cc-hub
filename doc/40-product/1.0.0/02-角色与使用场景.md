# 02-角色与使用场景

## Purpose
把 1.0.0 面向的角色、核心场景和关键用户路径写清楚，避免实现时偏向泛化能力而忽略主场景。

## Scope
本文件聚焦版本主用户和高价值路径，不扩展到团队级流程。

## Actors / Owners
- Owner: Product
- Readers: 产品、前端、后端、体验设计

## Inputs / Outputs
- Inputs: 问题定义、版本目标
- Outputs: Persona、JTBD、关键场景

## Core Concepts
- `P1 User`: 高频使用 Claude Code / Codex 的单人研发者。
- `P2 User`: 对 Agent 工作流治理有兴趣的技术负责人。
- `Primary Path`: 用户在 1.0.0 中最应该完成的主流程。

## Behavior / Flow
### Persona

| Persona | Description | Priority |
|---|---|---|
| `P1-Builder` | 已经高频使用 Agent，需要更强控制、回放和复用能力 | Highest |
| `P2-Lead` | 想把个人方法论沉淀成团队级规范，但当前仍以个人使用为主 | Secondary |

### JTBD

1. 当我把复杂任务交给 Agent 时，我希望能看到结构化时间线和失败拐点，这样我才能在关键时刻接管。
2. 当我验证某套 workflow / skill 生效时，我希望把它固化成可复用资产，而不是再靠记忆。
3. 当同一个目标拆成多个子任务时，我希望能在任务图里治理依赖、冲突和回写，而不是让复杂性淹没聊天框。

### High-Value Scenarios

| Scenario ID | Scenario | Why It Matters |
|---|---|---|
| `SC-001` | 单聊天会话发起复杂任务并观察实时事件 | 验证聊天主入口成立 |
| `SC-002` | 将复杂任务拆成 Task Graph 并为节点分配 AgentOS | 验证多任务治理成立 |
| `SC-003` | 生成回放文档并基于分析报告修订 SpecAsset | 验证证据闭环和调优闭环 |
| `SC-004` | 遇到权限请求、冲突或失败时由人接管并继续执行 | 验证治理价值成立 |

### Primary Path

1. 选择聊天 Agent，默认 `Claude Code`
2. 发起需求并创建 Session
3. 观察事件流，必要时切到 Task Graph 拆分任务
4. 等待任务完成或人工介入
5. 查看 Replay / Analysis
6. 将结论沉淀到 SpecAsset 并在下一次执行复用

## Interfaces / Types
- `Persona`
- `JTBD`
- `Scenario`
- `Primary Path`

## Failure Modes
- 如果实现只围绕“多 Agent 很强”，会丢掉单人高频用户的真正需求。
- 如果主路径不明确，GUI 会长成功能堆栈而不是工作流产品。

## Observability
- 所有主路径都应能在回放文档中复现:
  - 选择 Agent
  - 创建 Session
  - 任务拆分
  - 人工介入
  - 回放查看
  - Spec 修订

# 53-后端架构与输入输出规范

## Purpose
定义 CLAW 1.0.0 后端的结构边界、controller 层职责、输入输出模型和错误模型，作为 FastAPI / Pydantic 开发基线。

## Scope
本文件覆盖后端分层、Pydantic 输入输出规范、HTTP / WebSocket 响应结构和 controller 边界。

## Actors / Owners
- Owner: Product + Backend
- Readers: 后端、架构、前端接口对接者

## Inputs / Outputs
- Inputs: FastAPI / Pydantic 官方文档、当前 controller 索引
- Outputs: 后端目录基线、请求/响应模型、事件模型和错误模型

## Core Concepts
- `Input Model`: 面向请求校验的 Pydantic 模型
- `Output Model`: 面向响应输出的 Pydantic 模型
- `Internal Model`: 服务内部对象，不直接泄漏给 API
- `Event Envelope`: 面向事件流和 WebSocket 的统一信封
- `Controller`: 面向产品能力暴露的边界，不承载复杂业务编排

## Behavior / Flow
### Recommended Backend Structure

```text
backend/
  api/
    controllers/
    websocket/
  schemas/
    input/
    output/
    event/
    error/
  services/
  adapters/
  repositories/
  orchestrator/
  analysis/
```

### Layer Rules

| Layer | Responsibility | Must Not Do |
|---|---|---|
| `controller` | 接收请求、校验输入、调用 service、返回 output model | 不直接写复杂编排逻辑 |
| `service` | 业务流程和事务边界 | 不直接暴露 HTTP 语义 |
| `adapter` | 连接 Claude Code / Codex / 外部能力 | 不直接决定产品层状态 |
| `repository` | 持久化和读取 | 不承载业务规则 |

### FastAPI Controller Rules

基于 FastAPI 官方 `response_model` 和 `dependencies`：

- 所有对外接口都要定义明确的输入模型和输出模型
- `controller` 层只接收校验后的输入，不直接处理原始 dict
- 依赖注入统一使用 FastAPI `Depends`
- WebSocket 只负责连接和事件推送，不写复杂业务逻辑

### Pydantic Model Rules

基于 Pydantic 官方 models / validation / JSON schema：

- 请求入参统一使用 `InputModel`
- 响应出参统一使用 `OutputModel`
- 不把内部 domain object 直接作为响应返回
- 优先使用验证路径，不滥用 `model_construct()`
- 若输入本身是 JSON payload，优先沿用 JSON 验证路径

### I/O Structure

#### HTTP Success Response

```json
{
  "data": {},
  "meta": {
    "request_id": "req_xxx",
    "timestamp": "2026-04-19T12:00:00Z"
  }

```

```json
{
  "error": {

```

```json
{
  "event_id": "evt_xxx",

  "ts": "2026-04-19T12:00:00Z",
  "payload": {}
}
```

### Controller Baseline

1. `SessionController`
   - create / list / get / resume / stop
2. `ChatController`
   - submit input / interrupt / status
3. `TaskController`
   - create / update / delete / get
4. `TaskDependencyController`
   - add / remove / list dependencies
5. `WorkerController`
   - assign / status / retry / requeue / result
6. `EventController`
   - query / subscribe events
7. `ReplayController`
   - generate / get replay
8. `AnalysisController`
   - generate / get analysis
9. `SpecAssetController`
   - create / bind / list versions / revise
10. `GovernanceController`
   - permission decisions / conflicts / interventions

## Failure Modes
- controller 如果直接承载业务编排，后端会很快失控。
- 输入输出模型不分离，会把内部对象泄漏到 API 层。
- WebSocket 若直接推原始底层事件，前端会被底层差异污染。

## Observability
- 每个 controller 至少要暴露:
  - success rate
  - validation failure count
  - domain failure count
  - latency

## Open Questions / ADR Links
- 这里的 HTTP / WS envelope 是当前项目建议基线；最终字段命名可在 API 设计阶段微调，但结构原则不建议改。

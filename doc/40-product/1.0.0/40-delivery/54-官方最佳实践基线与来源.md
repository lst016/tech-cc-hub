# 54-官方最佳实践基线与来源

## Purpose
把 CLAW 1.0.0 采用的技术基线明确绑定到官方文档，确保开发规范不是拍脑袋，而是可追溯到权威来源。

## Scope
本文件覆盖 React、TypeScript、Tauri、FastAPI、Pydantic 五类官方来源。

## Actors / Owners
- Owner: Product + Architecture
- Readers: 前端、后端、架构、评审者

## Behavior / Flow
### Official Baseline Table

| Stack | Official Practice | CLAW Rule |
|---|---|---|
| React | 状态应避免矛盾、冗余、重复和深嵌套；refs 适合不驱动渲染的值 | Session / Task 只保留最小 UI 投影，临时句柄放 ref |
| TypeScript | 开启 `strict`、显式处理 `null` / `undefined` | 默认严格模式，拒绝把可空对象当确定值使用 |
| Tauri | 前端不要持有 secrets；特权能力应经过受控边界；权限和 scopes 由 runtime authority 控制 | 前端不直持 AgentOS secrets，未来若用 Tauri commands 必须经过 capability / scope |
| FastAPI | 用 `response_model` 约束输出；通过 `Depends` 建依赖；WebSocket 作为实时通道 | controller 必须分 input / output model，DI 走 `Depends`，事件流单独建模 |
| Pydantic | 优先使用验证路径；用 `model_dump` / `model_json_schema` / `model_validate*` 明确区分验证、序列化和 schema | 输入输出模型显式分离，JSON payload 优先走验证模型，不滥用无校验构造 |

### Project Interpretation Notes

- `React` 和 `TypeScript` 规则是直接工程基线
- `FastAPI` 和 `Pydantic` 规则直接决定后端 I/O 设计
- `Tauri` 规则更多影响前端安全边界和桌面壳层职责

这里有一部分是我根据官方文档对 CLAW 架构做的推导：
- “前端不持有 AgentOS secrets”
- “本地特权调用要走受控边界”
- “controller 不直接承载复杂编排逻辑”

这些属于基于官方资料和你现有架构文档做出的工程推论，不是官方原句。

## Failure Modes
- 如果没有来源基线，团队很容易把“个人偏好”误写成“项目规范”。
- 如果只贴链接不做项目化解释，开发期还是会回到自由发挥。

## Observability
- 未来如需升级技术栈，应先更新本文件，再改开发规范。

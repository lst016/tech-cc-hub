# 52-前端工程最佳实践

## Purpose
把 React、TypeScript、Tauri 的官方最佳实践收敛为 CLAW 前端开发基线，避免实现时各写各的。

## Scope
本文件覆盖状态建模、类型策略、前端安全边界和桌面壳层约束。

## Actors / Owners
- Owner: Product + Frontend
- Readers: 前端、架构、组件实现者

## Inputs / Outputs
- Inputs: 官方 React / TypeScript / Tauri 文档、当前 1.0.0 架构
- Outputs: 前端工程规则、状态边界和安全约束

## Core Concepts
- `Render State`: 真正影响 UI 呈现的状态
- `Ephemeral State`: 不应触发重渲染的临时值
- `Strict Types`: TypeScript 严格模式下的类型边界
- `Privileged Boundary`: 前端不能越过的能力边界
- `Copy Contract`: 用户可见文案也属于可验收的前端契约

## Behavior / Flow
### React State Rules

基于 React 官方关于状态结构和 escape hatches 的建议，项目内采用以下规则：

1. 只把真正影响渲染的内容放进 React / store state
2. 不把可计算值作为冗余 state 保存
3. 避免深层嵌套 state，优先扁平化
4. 不把 props 镜像进 state，除非明确是 `initial*`
5. `ref` 只用于不驱动渲染的值，例如 DOM、定时器、流句柄、滚动定位

对 CLAW 的具体落地：
- `selectedSessionId` 应存 ID，不存整份 Session 对象
- `selectedTaskId` 应存 ID，不存整份 TaskNode
- Timeline filters 可以是局部 state
- Session / Task / Event 的真相不应在多个组件里重复缓存

### Frontend State Ownership

| State Type | Owner |
|---|---|
| 页面临时 UI 状态 | 组件本地 state |
| 当前 Session / Task 选择 | 共享 store |
| 事件流、任务状态、回放摘要 | 后端 / 服务端真相 + 前端只读投影 |
| DOM / 非渲染引用 | `ref` |

### TypeScript Rules

基于 TypeScript 官方 `strict` / `strictNullChecks` 建议：

- `tsconfig` 必须开启 `strict`
- 默认不允许 `any`
- 对可空值必须显式建模
- 输入、输出、内部视图模型要分型，不混用
- 组件 props 不能依赖“隐式可选”

建议规则：
- API 模型使用 `Api*`
- UI 模型使用 `*ViewModel`
- 事件类型使用判别联合

### Tauri Boundary Rules

基于 Tauri 官方 process model、state management、runtime authority：

- 业务敏感逻辑和 secrets 不放在前端 webview
- 前端只拿到它需要的最小能力
- 任何本地特权调用都必须通过受控边界，而不是任意前端直达
- 若将来需要 Tauri command，必须受 capability / scope 约束

对 CLAW 的具体落地：
- 前端不直接持有外部 AgentOS secrets
- 前端不直接决定底层本地系统权限
- 前端只消费统一的 Session / Task / Event API

### Recommended Frontend Structure

```text
frontend/
  src/
    app/
    pages/
    components/
      chat/
      task-graph/
      timeline/
      replay/
      spec-assets/
      governance/
    stores/
    services/
    models/
```

### Recommended Engineering Gates

- 新组件必须有明确输入/输出状态
- 新 store 字段必须说明 owner 和来源
- 新 API 接入必须先有模型定义
- 新特权能力必须先过边界审查

### UI Copy Engineering Rule

前端实现必须把“UI 文案全中文”当作工程规则，而不是设计偏好：
- `1.0.0` 已交付界面的用户可见文案默认使用简体中文
- 不允许在组件中直接提交英文按钮文案、英文空态、英文提示语作为正式交付
- 允许保留英文的范围仅包括：品牌名、代码标识、接口字段、路径、事件类型、用户原始输入、第三方原始报错
- 若必须展示英文原始值，应在同一上下文补充中文说明，而不是只展示英文
- 组件评审和 QA 必须把文案语言一致性视为回归项

## Failure Modes
- 状态重复和深层嵌套会直接把复杂产品做崩。
- 前端越过 Tauri / 后端边界会让安全模型失效。
- 不开 strict 会让 Session / Task / Event 这种对象很快失控。

## Observability
- 前端实现评审时至少检查：
  - 是否镜像 props 到 state
  - 是否存在重复 Session / Task 真相
  - 是否绕过统一服务层

## Open Questions / ADR Links
- 当前规则是基于现有技术栈的默认基线；若未来前端框架变化，应重审本文件。

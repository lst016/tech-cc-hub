# 51-前端UI风格规范

## Purpose
定义 CLAW 1.0.0 的前端视觉风格、设计语言和 UI 约束，避免实现阶段退化成通用 AI 工具界面。

## Scope
本文件定义视觉方向、布局节奏、颜色、字体、状态和动效规则。
本文件不定义 Figma 像素稿，也不规定具体 CSS 实现方式。

## Actors / Owners
- Owner: Product
- Readers: 前端、设计、组件实现者

## Inputs / Outputs
- Inputs: 双主轴 IA、桌面优先场景、组件索引
- Outputs: UI 风格基线、视觉 token 方向、组件状态规范

## Core Concepts
- `Design Direction`: `Operator Console + Research Desk`
- `Desktop First`: 面向高频研发者的桌面密度，而不是移动端卡片化体验
- `Signal Colors`: 用少量高辨识度颜色表达运行状态，而不是把颜色当装饰
- `Readable Density`: 信息密度高，但不允许层级塌陷
- `Chinese First Copy`: 已交付 UI 文案默认使用简体中文，而不是中英混写

## Behavior / Flow
### Visual Direction

CLAW 的默认视觉应是：
- 专业、冷静、偏工程控制台，而不是聊天机器人玩具
- 有“工作台”感，而不是全屏单列聊天流
- 信息密集但层级清楚
- 明显区分 `Chat Axis` 和 `Task Graph Axis`

### Typography

主字体建议：
- UI Sans: `IBM Plex Sans`
- Mono / Code: `JetBrains Mono`

规则：
- 不使用 `Inter` 作为默认品牌感字体
- 标题用更紧凑、更有技术感的字重
- 数字、状态、事件 ID、路径统一用 monospace

### UI Copy Language

`1.0.0` 的界面文案规则：
- 所有面向用户的导航、按钮、提示、空态、表单标签、状态说明默认必须使用简体中文
- 不允许把英文占位文案、英文按钮名、英文空态直接交付到正式 UI
- 品牌名、产品名、Agent 名可以保留原文，例如 `Claude Code`、`Codex`
- 代码标识、路径、事件类型、协议字段、错误原文可保留英文，但需要与中文解释分层展示
- 如果未来要做多语言，中文仍然是默认语言包和验收基线

### Color System

基础色采用暖中性色 + 信号色，不用紫色系做默认品牌色。

建议语义色：

```text
--bg-canvas: #F4F1EA
--bg-panel: #FBF9F4
--bg-elevated: #FFFFFF
--line-soft: #D9D2C6
--text-primary: #1F252B
--text-secondary: #58636E
--signal-active: #0F766E
--signal-running: #1D4ED8
--signal-warning: #B45309
--signal-danger: #B42318
--signal-spec: #7C3AED
```

约束：
- 不允许在组件里直接硬编码业务颜色
- 所有状态色必须走语义 token
- `SpecAsset` 可以使用紫罗兰系作为辅助识别，但不能让全站变成紫色主调

### Layout Rules

- 页面主结构固定为 `左侧导航 + 中央主工作区 + 右侧上下文面板 + 底部可收纳产物区`
- Chat 和 Task Graph 都必须是一等视图，不允许其中一个被弱化为弹层
- 面板边界清楚，允许细线、浅阴影和轻微层级差
- 列表、时间线、节点卡片优先使用“信息板”式布局，不做大面积圆角卡片堆叠

### Component Shape Language

- 圆角偏小，建议 `6px ~ 10px`
- 边框和分隔优先于重阴影
- 状态通过 `标签 + 色条 + 图标` 联合表达
- 重点按钮克制，主 CTA 不应在一个屏幕出现过多

### Motion

- 动效只服务于状态理解，不服务于炫技
- 允许：
  - 面板切换淡入
  - 时间线新增的轻量高亮
  - 节点状态变化的短反馈
- 不允许：
  - 全局漂浮动画
  - 长时间 skeleton 占主视野
  - 花哨缩放 / bounce

### Status Design

关键状态必须有统一视觉语言：

| State | Visual Rule |
|---|---|
| `running` | 蓝色 / 青色强调 + 进行中 icon |
| `completed` | 低饱和成功色 + 完成标记 |
| `blocked` | 暖色强调 + 原因入口 |
| `failed` | 红色强调 + 错误摘要入口 |
| `requires_attention` | 明显但不刺眼的 warning tag |

### Accessibility

- 文本与背景必须保持足够对比
- 状态不可只靠颜色表达
- 键盘焦点必须明显
- 时间线、任务图和侧栏都要保留清晰 focus ring

## Failure Modes
- 如果前端风格走成普通聊天机器人 UI，产品定位会立刻被削弱。
- 如果颜色和动效没有语义，复杂状态会更难理解而不是更清楚。
- 如果交付界面中英混杂，产品完成度和本地化一致性会明显下降。

## Observability
- 设计验收要至少抽查：
  - Chat Workspace
  - Task Graph Workspace
  - Live Timeline
  - Replay / Analysis
- 文案验收要抽查：
  - 顶级导航
  - 主按钮与次按钮
  - 空状态
  - 错误提示
  - Menu / Drawer / Panel 标题

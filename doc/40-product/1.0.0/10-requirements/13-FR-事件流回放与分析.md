# 13-FR-事件流回放与分析

## Purpose
定义 Evidence Loop 在 1.0.0 的核心产品能力。

## Scope
本文件覆盖事件入流、时间线、ReplayDocument 和 AnalysisReport。

## Actors / Owners
- Owner: Product
- Readers: 后端、前端、分析实现者

## Inputs / Outputs
- Inputs: EventEnvelope、Session、TaskNode
- Outputs: `FR-EVID-*`

## Interfaces / Types
`FR-EVID-001: MUST - 系统必须将底层 AgentOS 行为归一为统一事件流`
Acceptance Criteria:
- 至少支持 Session、工具、权限、任务、文件变化五类事件
- 每个事件都带时间戳和归属信息
- 事件写入失败必须可见

`FR-EVID-002: MUST - 用户可以看到实时时间线`
Acceptance Criteria:
- Live Timeline 至少展示事件类型、时间、归属对象
- 用户可从时间线跳转到关联 Session / Task / Replay
- 时间线支持按对象过滤

`FR-EVID-003: MUST - 系统能为一次复杂任务生成 ReplayDocument`
Acceptance Criteria:
- Replay 至少包含任务目标、关键转折、人工介入和结果
- Replay 可从 Session 或 Task 打开
- Replay 生成失败时必须给出原因

`FR-EVID-004: MUST - 系统能生成基础 AnalysisReport`
Acceptance Criteria:
- 报告至少包含耗时、事件量、人工介入、失败次数
- 报告能回链到原始 Session 或 Task
- 报告生成与否有明确状态

`FR-EVID-005: SHOULD - 用户可以对比同类任务的两次运行结果`
Acceptance Criteria:
- 至少支持本次与上次运行的对比
- 对比维度至少包含人工介入率、失败次数、总耗时
- 对比结果可回链到对应 Replay

`FR-EVID-006: SHOULD - 系统可以标记回放覆盖率不足的任务`
Acceptance Criteria:
- 任务结束后可判断是否缺关键事件
- 缺口原因至少分为 `missing events / partial write / unsupported capability`
- 覆盖率不足不应被误标为成功闭环

## Failure Modes
- 如果只有日志没有 Replay，用户仍然需要自己拼接上下文。
- 如果 Analysis 不能回链到原始事件，指标会失去可信度。

## Observability
- 必须记录:
  - `event_normalized`
  - `timeline_opened`
  - `replay_generated`
  - `analysis_generated`
  - `replay_compare_requested`

# 16-FR-权限冲突与人工介入

## Purpose
定义人工接管、权限请求和冲突治理在 1.0.0 中的功能要求。

## Scope
本文件覆盖请求审批、冲突列表、人工决策记录和恢复执行。

## Actors / Owners
- Owner: Product
- Readers: 前端、后端、架构、测试

## Inputs / Outputs
- Inputs: 权限模型、冲突处理规范
- Outputs: `FR-GOV-*`

## Interfaces / Types
`FR-GOV-001: MUST - 系统必须向用户呈现权限请求并允许批准或拒绝`
Acceptance Criteria:
- 至少展示请求来源、目标动作和影响范围
- 用户可以批准或拒绝
- 决策结果进入事件流

`FR-GOV-002: MUST - 系统必须收敛并展示冲突项`
Acceptance Criteria:
- 至少支持文件冲突、版本冲突和调度冲突
- 冲突项必须可从任务或回放入口进入
- 未处理冲突不能被静默忽略

`FR-GOV-003: MUST - 用户的人工介入必须被记录为正式证据`
Acceptance Criteria:
- 至少记录介入时间、对象、动作和结果
- 人工介入可从 Replay 中看到
- AnalysisReport 可统计人工介入次数

`FR-GOV-004: SHOULD - 用户可以在处理权限或冲突后继续执行当前流程`
Acceptance Criteria:
- 决策完成后可恢复 Session 或 Task
- 恢复动作有明确反馈
- 恢复失败时说明原因

`FR-GOV-005: SHOULD - 系统支持最小化重复审批`
Acceptance Criteria:
- 对相同来源和同类动作可提供历史上下文
- 1.0.0 不要求全自动政策执行
- 不能因为减少重复审批而绕开显式记录

`FR-GOV-006: MUST - 人工治理不能破坏证据链`
Acceptance Criteria:
- 任何人工动作都必须可回放
- 任何人工动作都能回链到 Session / Task
- 人工动作不允许只存在于界面瞬时状态

## Failure Modes
- 如果权限或冲突处理不落证据，CLAW 的治理价值会直接失效。
- 如果人工介入后不能恢复流程，用户仍然会退回终端。

## Observability
- 必须记录:
  - `permission_requested`
  - `permission_decided`
  - `conflict_detected`
  - `conflict_resolved`
  - `human_intervened`

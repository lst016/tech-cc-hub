# 11-FR-聊天与会话控制

## Purpose
定义聊天主入口与会话管理在 1.0.0 中必须提供的产品能力。

## Scope
本文件覆盖聊天工作台、Session 生命周期和交互主 Agent 选择。

## Actors / Owners
- Owner: Product
- Readers: 前端、后端、架构、测试

## Inputs / Outputs
- Inputs: Primary Path、AgentOS 集成约束
- Outputs: `FR-CHAT-*`

## Interfaces / Types
### Requirements

`FR-CHAT-001: MUST - 用户能在聊天界面选择 Claude Code 或 Codex，且默认选中 Claude Code`
Acceptance Criteria:
- 新建 Session 时默认显示 `Claude Code`
- 选项仅允许 `Claude Code` 与 `Codex`
- 同一时刻只能有一个活跃聊天 Agent
- 切换聊天 Agent 会形成可回放事件

`FR-CHAT-002: MUST - 用户可以创建、命名并进入一个新的聊天 Session`
Acceptance Criteria:
- 新建 Session 后立即可发送第一条消息
- Session 至少记录标题、创建时间、当前活跃 Agent
- Session 在侧边栏可再次打开

`FR-CHAT-003: MUST - 用户在 Session 中发送输入后，界面能展示流式响应和执行过程`
Acceptance Criteria:
- 用户发送后不需要刷新页面即可看到响应
- 执行中的事件需进入 Live Timeline
- 失败、停止和完成状态必须有明确 UI 反馈

`FR-CHAT-004: MUST - 用户可以主动中断当前聊天执行`
Acceptance Criteria:
- 中断操作有明确入口
- 中断成功后 Session 状态同步更新
- 中断动作写入事件流

`FR-CHAT-005: SHOULD - 用户可以恢复最近关闭或暂离的 Session`
Acceptance Criteria:
- 返回 Session 后可继续查看历史事件和消息
- 恢复后保持上一次活跃 Agent 绑定
- 若底层 AgentOS 不支持继续执行，系统需给出可理解提示

`FR-CHAT-006: SHOULD - 聊天界面可展示当前 Session 关联的任务、Spec 资产和回放入口`
Acceptance Criteria:
- Session 详情至少显示关联 Task 数量
- 若绑定 SpecAsset，应可跳转查看
- 若已生成 Replay / Analysis，应可直接打开

## Failure Modes
- 如果聊天会话不记录主 Agent，后续回放会丢失路由语义。
- 如果聊天入口允许多 Agent 同时激活，会破坏消息归属和证据链。

## Observability
- 必须记录:
  - `chat_agent_selected`
  - `session_created`
  - `user_input_submitted`
  - `session_interrupted`
  - `session_resumed`

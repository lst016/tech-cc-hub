# 13-执行可观测层-资源附录

## Purpose
这个附录不是单纯的链接仓库，而是给 `13-执行可观测层` 提供一个可持续扩展的资源地图。后面继续“抄”时，优先从这里沿着官方 URL 往下深挖，不需要重新散搜。

## Resource Landscape
外部资源大体可以拆成四类：

1. `对象模型资源`
定义 trace、span、thread、session、run、call、observation 这些核心对象。

2. `产品形态资源`
提供 trace tree、timeline、detail drawer、session replay、agent graph、long text viewer 这些 UI 形态。

3. `指标与聚合资源`
提供 latency、token、cost、RED、会话级聚合、跨 trace 指标等口径。

4. `标准与协议资源`
提供 GenAI attributes、MCP span naming、AI span kind taxonomy 等标准化字段。

## G. 资源里已经暴露出来的几个强信号

1. `Session` 必做
LangSmith、Langfuse、Phoenix、W&B 都有“多轮归组”概念。说明如果只做单次 trace，我们会很快遇到上限。

2. `Observation-centric` 比单 trace blob 更适合分析
Langfuse 已经把模型推进到 observation-centric data model，这对我们很有启发。以后如果想做筛选、聚合、导出，观察对象必须是一级公民。

3. `Tool provenance` 很值钱
ADK BigQuery Agent Analytics 直接把本地工具、MCP、子 agent、A2A 远程 agent 分开，这比简单显示 `toolName` 有更强的分析价值。

4. `Long text viewer` 不是加分项，是必需项
W&B 直接把长文本查看当卖点。我们的节点详情如果还让用户看折叠后的纯 textarea，会很吃亏。

5. `Trace -> Eval` 应该从一开始预埋
OpenAI、Phoenix、LangSmith 都说明可观测层最终会流向评估和回归，而不是止于看日志。

## H. 对我们最有价值的外部字段清单
这些字段建议优先映射到我们的内部模型中：

| 外部字段 | 建议映射到我们哪里 | 来源 |
|---|---|---|
| `trace_id` | `trace.traceId` | OpenAI / OTel / ADK |
| `group_id` | `trace.sessionId` 或 `threadId` | OpenAI |
| `sessionId` | `session.id` | Langfuse |
| `run` / `observation` / `call` | `node` | LangSmith / Langfuse / Weave |
| `openinference.span.kind` | `node.kind` | OpenInference |
| `gen_ai.input.messages` | `context.userPrompt` / `context.inputMessages` | OTel |
| `gen_ai.system_instructions` | `context.systemInstructions` | OTel |
| `gen_ai.retrieval.documents` | `context.retrievalDocs` | OTel |
| `gen_ai.tool.call.arguments` | `detail.inputStructured` | OTel |
| `gen_ai.tool.call.result` | `detail.outputStructured` | OTel |
| `trace_id` + `span_id` | `node.traceId` + `node.spanId` | ADK / OTel |
| `tool provenance` | `node.provenance` | ADK |

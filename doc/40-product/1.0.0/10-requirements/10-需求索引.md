# 10-需求索引

## Purpose
建立 1.0.0 的功能需求目录和编号规则，让需求可以按主题不断拆分，而不破坏全局可追踪性。

## Scope
本文件只定义 FR 目录、ID 前缀和阅读顺序。

## Actors / Owners
- Owner: Product
- Readers: 产品、前后端、测试、项目管理

## Inputs / Outputs
- Inputs: 版本范围、核心场景
- Outputs: 需求拆分结构、ID 规则

## Core Concepts
- `FR`: Functional Requirement
- `NFR`: Non-Functional Requirement
- `Requirement Area`: 功能主题拆分单元

### Rules

1. 新需求必须进入对应主题文档。
2. 如果某主题增长过快，应继续拆分为子文档，而不是回到总表。
3. 所有 FR 必须带:
   - 唯一 ID
   - MoSCoW 优先级
   - 3 条以上验收标准

## Interfaces / Types
- `FR-CHAT-xxx`
- `FR-GRAPH-xxx`
- `FR-EVID-xxx`
- `FR-SPEC-xxx`
- `FR-WS-xxx`
- `FR-GOV-xxx`

## Failure Modes
- 如果 FR ID 没有主题前缀，后续追踪矩阵会很难维护。
- 如果需求只保留总表，不分主题，后续会再次膨胀成大文件。

## Observability
- 每个需求文档都必须能映射到至少一个 Epic 和一个交付文档。

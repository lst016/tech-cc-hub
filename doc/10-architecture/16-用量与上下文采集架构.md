# 用量与上下文采集架构

## 1. 概述

本文档描述 tech-cc-hub 中"用量数据"的完整采集链路。用量数据包含两个维度：

| 维度 | 说明 | 典型字段 |
|------|------|---------|
| **执行用量** | Agent 每次运行的资源消耗 | input/output tokens、消耗金额(USD)、耗时(ms) |
| **上下文构成** | 发送给模型的 Prompt 由哪些部分组成，每部分大小 | buckets/segments 结构，按来源分类的字符数和 token 估算 |

两者互相独立但互补：执行用量回答"花了多少"，上下文构成回答"花在哪里"。

---

## 2. 执行用量采集

### 2.1 数据来源

执行用量的原始数据来自 **Claude Agent SDK** 的 `SDKResultMessage`。SDK 在每次 `query()` 调用完成后返回该消息，包含：

```
SDKResultMessage
 ├── duration_ms          // 总运行时长（毫秒）
 ├── total_cost_usd       // 总估算成本（美元）
 ├── usage
 │    ├── input_tokens
 │    ├── output_tokens
 │    ├── cache_read_input_tokens
 │    ├── cache_creation_input_tokens
 │    └── web_search_requests
 ├── modelUsage           // 按模型细分的用量（Record<string, ModelUsage>）
 └── num_turns            // 交互轮数
```

> 引用：`node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts` L3119-3162

### 2.2 主链路：会话消息流

这是用量采集的**主路径**，覆盖每一次聊天会话的执行。

```
┌──────────────────┐
│  Claude Agent SDK │  query() 执行完成后 yield SDKResultMessage
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  runner.ts       │  runClaude() — 通过 for await (const message of q) 接收
│  (src/electron/  │  将每条 SDKMessage 通过 onEvent() 广播出去
│   libs/runner.ts)│
└────────┬─────────┘
         │  onEvent({ type: "stream.message", payload: { sessionId, message } })
         ▼
┌──────────────────┐
│  ipc-handlers.ts │  emit() 函数 — 两条并行路径：
│  (src/electron/  │
│   ipc-handlers.ts│  ┌─► session-store.recordMessage() → SQLite sessions.db
│   L423-492)       │  │   messages 表 (id, session_id, data JSON, created_at)
│                   │  │   data 列包含完整的 SDKResultMessage（含 usage 字段）
│                   │  │
│                   │  └─► broadcast() → webContents.send("server-event", ...)
│                   │       → 渲染进程接收后用于实时 UI 展示
└──────────────────┘
```

**代码引用**：

- `runner.ts` L438-458：`for await` 循环 yield SDKMessage，L449 检测到 `message.type === "result"` 时发送 `session.status`。
- `ipc-handlers.ts` L423-492：`emit()` 函数，先 persist 再 broadcast。
- `session-store.ts` L413-434：`recordMessage()` 将消息 JSON 写入 SQLite。

### 2.3 子链路：任务系统用量

任务系统（/task 功能）独立跟踪每个执行任务的 token 和成本。

```
┌──────────────────┐
│  task/executor.ts │  extractUsage() — 从每条 SDK 消息中扫描 usage 字段
│  L772-786         │  查找 input_tokens / output_tokens / total_cost_usd
└────────┬─────────┘
         │  每次检测到新用量时调用 repo.recordUsage()
         ▼
┌──────────────────┐
│  task/repository │  更新 tasks 和 task_executions 表的
```

### 2.4 UI 侧用量提取

渲染进程通过 `ActivityRailModel` 从已持久化的会话消息中重建用量指标。

```
src/shared/activity-rail-model.ts L1968-1975:

if (message.type === "result") {
  const result = message as SDKResultMessage;
  latestDurationMs = result.duration_ms ?? latestDurationMs;
  latestInputTokens = result.usage?.input_tokens ?? latestInputTokens;
  latestOutputTokens = result.usage?.output_tokens ?? latestOutputTokens;
  latestCostUsd = result.total_cost_usd ?? latestCostUsd;
}
```

这些值聚合到 `ActivityRailModel.summary`（L2642-2683），产出：
- `durationLabel` — 格式化耗时
- `inputLabel` / `contextLabel` / `outputLabel` — 字符和 token 计数
- `costLabel` — 格式化 USD 成本
- `successCount` / `failureCount`

## 3. 上下文构成采集（Prompt Ledger）

### 3.1 设计目的

每一次 `session.start` / `session.continue` 请求发送到 SDK 之前，系统会分析即将发送的 Prompt 的**组成结构**，生成一张"构成清单"。这张清单回答：

- 总字符数和估算 token 数是多少？
- 系统指令占多少？历史对话占多少？用户输入占多少？
- 哪些部分可能存在优化空间（过长、歧义引用、工具载荷过大）？

### 3.2 数据模型

```
PromptLedgerMessage
 ├── type: "prompt_ledger"
 ├── phase: "start" | "continue"
 ├── model / cwd / capturedAt
 ├── totalChars              // 总字符数
 ├── totalTokenEstimate      // 本地估算总 token 数
 ├── buckets[]               // 按来源分组的汇总
 │    ├── sourceKind          // system / project / skill / workflow / current /
 │    │                       // attachment / memory / history / tool / other
 │    ├── chars / tokenEstimate / ratio
 │    └── sample              // 内容摘要
 └── segments[]              // 精细切分到每个独立段
      ├── sourceKind / segmentKind
      ├── chars / tokenEstimate / ratio / sample
      ├── risks[]             // long_content / repeated_content /
      │                       // ambiguous_reference / missing_acceptance /
      │                       // tool_payload
      └── optimizationHint    // 优化建议文本
```

> 完整类型定义：`src/shared/prompt-ledger.ts` L78-89

### 3.3 Token 估算算法

```typescript
// src/shared/prompt-ledger.ts L120-137
function estimatePromptLedgerTokens(text: string): number {
  // CJK 字符 × 1.2
  // 非 CJK 非空白字符 × 1/3
  // 空白字符 × 0.15
}
```

这不是 SDK 报告的真实 token 数，而是本地快速估算，用于在发送前提供"预计花费多少上下文"的参考。

### 3.4 采集时机

`ipc-handlers.ts` L365-421 — `buildPromptLedgerForRun()`：

```
session.start / session.continue 处理函数
  │
  ├──► 收集 promptSources（系统指令 + 项目规则 + 技能）
  ├──► 收集 memorySources（延续摘要）
  ├──► 收集 historyMessages（历史会话）
  ├──► 收集 attachments（附件）
  │
  └──► buildPromptLedgerMessage() → 产出 PromptLedgerMessage
         │
         └──► 作为 stream.message 事件发送，与普通消息同一通道
```

### 3.5 分段与风险分析

`PromptLedgerSegment` 除了携带字符数和 token 估算，还包含：

**风险标记**（`prompt-ledger.ts` L421-438）：
- `long_content` — 单段 ≥4000 字符
- `ambiguous_reference` — 短内容中包含指代不明的代词（它/他/她/这个/那个）
- `missing_acceptance` — 当前用户输入未包含验收标准关键词
- `tool_payload` — 工具输入/输出内容

**优化建议**（`prompt-ledger.ts` L441-458）：
- 工具返回较长 → 建议压缩为摘要
- 历史内容偏长 → 建议折叠为最近决策
- 当前输入 → 建议补齐目标和验收标准

### 3.6 上下文可视化

Prompt Ledger 的数据在 UI 中通过两个工具函数展示：

| 文件 | 作用 |
|------|------|
| `src/ui/utils/context-usage-cells.ts` | 将 token 分布渲染为固定大小（默认 40 格）的彩色单元格网格 |
| `src/ui/utils/context-usage-breakdown.ts` | 按分类过滤段，生成供 SessionAnalysisPage 使用的明细列表 |

## 5. 数据存储汇总

| 存储 | 表 | 用量字段 | 粒度 |
|------|-----|---------|------|
| `sessions.db` | `messages.data` (JSON) | `usage.input_tokens`, `usage.output_tokens`, `total_cost_usd`, `duration_ms` | 每次 SDK query |
| `tasks.db` | `tasks` | `input_tokens`, `output_tokens`, `estimated_cost_usd` | 每次任务执行 |
| `tasks.db` | `task_executions` | `input_tokens`, `output_tokens`, `estimated_cost_usd` | 每次执行记录 |
| 无（运行时） | PromptLedgerMessage | `totalChars`, `totalTokenEstimate`, `buckets[]` | session start/continue 时生成，随消息流广播 |

## 6. 关键设计决策

### 6.1 用量数据不单独建表

会话的用量数据**没有独立的 usage 表**，而是嵌入在 `messages.data` JSON 字段中。原因是：
- 用量数据始终伴随 SDK 消息存在，分离存储会导致双写一致性问题
- 查询用量时总是从消息历史重建，不需要独立索引
- 任务系统因为需要聚合统计（SUM cost、按任务归集），所以单独列了 `input_tokens` / `output_tokens` / `estimated_cost_usd` 列

### 6.2 Token 估算不等于 SDK 计数

Prompt Ledger 的 `totalTokenEstimate` 使用简单的字符加权公式，**不是** SDK 返回的真实 token 数。两者用途不同：

| | Prompt Ledger 估算 | SDK 报告 |
|---|---|---|
| 时机 | 请求发送前 | 请求完成后 |
| 算法 | 字符加权公式 | 模型 tokenizer |
| 用途 | 提示优化、上下文构成分析 | 成本核算、用量展示 |
| 精度 | ±30-50% | 精确 |

### 6.3 两套用量采集相互独立

聊天会话的用量（runner → session-store）和任务系统的用量（executor → repo.recordUsage）走不同的持久化路径。两者都从 SDK 消息中提取数据，但互不依赖。

# AI 接口与模型目录设计

## 目标

把现有纵向堆叠的 AI 接口配置拆成三个工作面：接口连接、模型目录、路由策略。界面直接使用网关实际拉取的模型数据，支持上百模型的搜索、能力筛选、分页、批量纳管与单模型配置，同时保持现有 Anthropic 兼容运行协议不变。

## 信息架构

- **接口连接**：左侧连接列表，右侧仅编辑当前连接；保留新增、启停、密钥、测试、同步模型、Codex OAuth 等现有能力。
- **模型目录**：每一行代表一个“网关部署”，唯一键为 `profileId + modelName`。同名模型位于不同网关时必须显示为不同记录。
- **路由策略**：继续配置主模型、专家模型、小模型、分析模型、图片理解和图片生成，但候选模型只来自已纳管目录。

## 目录数据契约

模型发现信息来自网关返回：模型 ID、`ownedBy`、`supportedEndpointTypes`、`createdAt`。本地仅补充别名、标签、上下文、压缩阈值、路由优先级和纳管状态，不伪造价格、健康、同步时间。

- 旧配置没有纳管字段时视为已纳管，保证升级兼容。
- 同步发现的新模型默认纳管；历史“待纳管”状态在加载时自动迁移为已纳管，只有用户明确排除的模型继续留在可用池之外。
- 排除正在被路由槽使用的模型必须被阻止并说明原因。
- 能力分类优先使用上游 endpoint metadata；只有上游未提供 metadata 时才按模型名称回退识别。
- `routingWeight` 表示同名模型跨网关选择时的路由优先级，不表达流量百分比。

## 自动同步策略

- 软件启动并完成界面加载后自动刷新一次已启用连接，运行期间每 24 小时再刷新一次。
- 自动刷新采用追加式合并：新增模型默认纳管，保留用户排除状态和本地覆盖字段，不因网关一次漏返而删除已有模型，也不自动改动任何路由槽。
- 单个连接失败不阻塞其他连接；请求 45 秒超时，退出时中止，且拉取完成后会基于最新配置再次核对连接，避免覆盖同时发生的手工修改。
- 只有成功保存了真正新增的模型时才聚合弹出一次应用内通知；重复模型、纯 metadata 更新或失败不打扰用户。
- 设置窗口存在未保存修改时，只把后台新增模型合入当前草稿，保留用户编辑和显式删除。

## 交互与布局

设置侧栏保持现状，AI 接口页内部增加三级标签。模型目录桌面布局为筛选栏 + 表格 + 右侧详情面板；每页 20 条，避免一次渲染全部模型。批量操作只影响勾选的部署行。右侧详情只开放本地覆盖字段，发现字段只读。

接口连接采用约 340px 的连接列表和自适应详情区。API Key 默认密码显示并提供显隐控制。移动宽度下主从区域顺序堆叠，不依赖横向滚动完成主要操作。

## 验证

- 纯函数测试覆盖唯一键、能力分类、组合筛选、批量纳管、详情编辑、不可排除路由中模型与不可变更新。
- 配置存取测试覆盖目录字段持久化及旧配置默认语义。
- 构建、类型检查、针对性测试和静态检查通过。
- 使用参考图与实现截图做同 viewport 对照；每轮通过 visual-verdict，最终结果写入 `design-qa.md`。

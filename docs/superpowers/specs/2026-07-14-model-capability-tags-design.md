# 模型多能力标签设计

## 背景

模型目录当前把模型能力推断为单一类别。网关实际拉取的很多模型只有模型 ID，没有 `supportedEndpointTypes`，因此 GPT-5.6、MiniMax-M3 等同时支持文本和图片理解的模型容易被归为纯文本或仅归为图片理解。

能力不是互斥分类。文本、推理和图片理解可以同时存在；模型目录和路由策略应使用同一组多能力标签。

## 目标

- 使用类型化数组保存模型的本地能力覆盖。
- 自动推断返回多能力标签，而不是单一类别。
- MiniMax-M3 默认识别为文本与图片理解；MiniMax M2.x 暂时只识别为文本。
- 网关重新同步模型时保留本地人工覆盖。
- 旧配置无需迁移即可继续加载。

## 数据结构

在单个模型配置上新增可选字段：

```ts
capabilities?: Array<
  | "text"
  | "reasoning"
  | "image-understanding"
  | "image-generation"
  | "embedding"
  | "rerank"
  | "audio"
>;
```

字段缺失表示自动识别；非空数组表示本地人工覆盖。保存时只接受已知值、去重并保持稳定顺序。空数组规范化为字段缺失，避免出现“模型没有任何能力”的无效状态。

不复用 `supportedEndpointTypes`，因为它描述上游协议或端点；不复用自由文本 `tags`，因为它不能为路由提供可靠的类型约束。

## 有效能力计算

优先级如下：

1. 存在本地 `capabilities` 时直接使用本地覆盖。
2. 否则读取上游 `supportedEndpointTypes`。
3. 上游未提供完整能力时使用共享模型族识别器补充。
4. 无法识别的普通对话模型默认具有 `text`。

自动推断规则：

- 嵌入、重排、音频和纯生图模型保持对应的专用能力。
- 普通对话模型先加入 `text`。
- 命中推理规则时追加 `reasoning`。
- 命中图片理解端点或共享模型族规则时追加 `image-understanding`。
- `MiniMax-M3` 自动得到 `["text", "image-understanding"]`。
- `MiniMax-M2`、`MiniMax-M2.1`、`MiniMax-M2.5`、`MiniMax-M2.7` 及 highspeed 版本暂时得到 `["text"]`。

## 同步与配置兼容

- 模型同步合并上游元数据时保留现有 `capabilities`。
- Electron 配置层、共享类型和 UI 类型使用同一个能力联合类型。
- 配置读取时过滤未知能力值；不修改旧模型的其他字段。
- 旧配置没有 `capabilities` 时继续走自动识别，因此没有破坏性迁移。

## UI 与路由

- 模型目录表格展示最多三个有效能力标签。
- 模型详情的“本地配置”提供能力标签开关。
- 自动识别状态显示“自动”；修改任一标签后切换为“本地覆盖”。
- 提供“恢复自动识别”，删除持久化的 `capabilities` 字段。
- 图片预处理、生图等路由候选直接按有效能力数组过滤。

## 验证

- 单元测试覆盖多标签推断、MiniMax-M3、MiniMax M2.x 和人工覆盖优先级。
- 配置测试覆盖未知值过滤、去重、旧配置兼容和同步保留覆盖。
- UI 测试覆盖能力开关、恢复自动识别和路由下拉实时更新。
- 使用本机脱敏后的 133 条模型目录验证 MiniMax-M3 与 GPT-5.6 均出现在图片理解候选中。

## 非目标

- 本次不自动把所有 MiniMax M2.x 认定为多模态。
- 本次不从第三方在线模型数据库动态同步能力。
- 本次不改变网关接口协议或模型调用方式。

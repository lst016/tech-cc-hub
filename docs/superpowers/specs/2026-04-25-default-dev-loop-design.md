# 默认 Dev Loop：开发任务自动验证与视觉复刻闭环设计

## 背景

用户希望 `tech-cc-hub` 在开发任务上具备接近 Cursor 的体验：不是让 Agent 一次性猜准，而是系统默认建立“执行、预览、验证、修复”的反馈闭环。尤其在图片、Figma、截图、页面复刻和 UI 调整任务里，当前 Agent 经常需要用户反复指出偏差；Cursor 的优势主要来自更紧密的视觉反馈循环，而不是单次生成能力绝对更强。

`tech-cc-hub` 已经具备 Agent 执行、会话记录、Trace Viewer、Prompt Ledger、图片摘要、原图回看和 slash command 能力。缺失的是一个系统级的默认开发任务策略：当任务被识别为开发类任务时，自动选择合适的验证闭环；当任务涉及 UI/图片/Figma 时，自动进入视觉开发闭环。

## 产品目标

1. 只要识别为开发任务，默认进入 Dev Loop，不再依赖用户手动提醒“跑一下”“验证一下”“截图看一下”。
2. 根据任务类型自动选择验证闭环：
   - 后端、逻辑、API、脚本任务走测试/构建闭环。
   - 前端、UI、Figma、截图、页面复刻任务走视觉开发闭环。
   - Electron 桌面端任务走真窗口闭环。
3. 对视觉类任务，系统自动完成预览、截图、视觉比对、偏差清单、修复建议和再验证。
4. Trace Viewer 能记录每轮闭环证据，包括目标图、当前截图、差异、修复动作、验证结果。
5. 用户仍可关闭或降级闭环，避免简单文案修改被过度验证。

## 非目标

1. 本轮设计不要求替换模型或重写 Agent SDK。
2. 不要求所有开发任务都截图；截图只对视觉/前端/桌面端任务默认启用。
3. 不要求一次性实现全自动多轮复刻；可以先实现最小可用闭环，再扩展到多轮自动修复。
4. 不要求完全复刻 Cursor 的交互外观，重点是复刻反馈机制。

## 任务识别策略

系统应在用户提交任务后先做轻量分类，得到 `taskKind` 和 `loopMode`。

### taskKind

1. `code`
   - 普通代码修改、bug 修复、重构、脚本、后端接口、数据处理。

2. `frontend`
   - React、Vue、CSS、Tailwind、组件、页面、布局、交互、样式。

3. `visual`
   - 用户上传图片、截图、Figma，或文本包含“复刻、对齐、按图、UI、视觉、页面、样式、布局、像设计稿”等信号。

4. `electron`
   - 明确涉及桌面客户端、窗口、右栏、Electron 真窗口、截图验收。

5. `docs`
   - 文档、说明、计划、规范类任务，一般不进入代码验证闭环。

### loopMode

1. `none`
   - 非开发任务、纯问答、简单说明。

2. `dev`
   - 默认开发闭环：修改代码后必须运行测试、构建或项目内验证命令。

3. `visual-dev`
   - 视觉开发闭环：在 `dev` 基础上增加启动预览、截图、视觉比对、偏差修复。

4. `electron-window`
   - Electron 真窗口闭环：启动 Electron，窗口截图，检查真实客户端状态。

## 默认规则

1. 如果 `taskKind=code`，默认 `loopMode=dev`。
2. 如果 `taskKind=frontend`，默认 `loopMode=visual-dev`，除非没有页面预览入口。
3. 如果 `taskKind=visual`，强制 `loopMode=visual-dev`。
4. 如果 `taskKind=electron`，默认 `loopMode=electron-window`。
5. 如果用户明确说“只改文案、不用跑”“先别启动”，可以临时降级，但 Trace 中要记录降级原因。

## Dev Loop 工作流

通用开发任务默认走以下流程：

1. 需求理解
   - 识别目标、限制、验收标准。
   - 读取项目说明、脚本、相关文件。

2. 计划生成
   - 给 Agent 注入短计划要求。
   - 对复杂任务要求先写开发文档或实现计划。

3. 代码修改
   - 按项目现有模式改动。
   - 避免无关重构。

4. 验证执行
   - 自动选择测试、构建、lint、启动、接口调用等验证命令。
   - 失败时把错误作为下一轮输入。

5. 修复循环
   - 最少修复一轮明确失败。
   - 达到最大轮次或需要人工判断时暂停。

6. 证据输出
   - 最终回复必须包含运行过的命令和结果。
   - Trace Viewer 记录每轮验证状态。

## Visual Dev Loop 工作流

视觉开发任务默认走以下流程：

### First-Shot Design Pack

为了提高第一轮产出准确性，视觉类任务不能直接进入“看图写代码”。系统应先把目标视觉与当前项目状态整理成一份第一轮设计包，再让 Agent 修改代码。

设计包至少包含：

1. 目标图规格
   - 页面尺寸、主区域分块、颜色 token、字号/字重、间距、圆角、阴影、组件层级、滚动/固定区域。

2. 当前项目约束
   - 当前组件入口、现有 CSS/Tailwind 主题、可复用组件、不应改动的区域。

3. 映射关系
   - 目标规格对应到哪个组件、哪个容器、哪些样式变量或 class。

4. 第一轮验收标准
   - 主布局相似、关键模块首屏可见、文字不溢出、不遮挡、交互状态正确。

5. 禁止行为
   - 缺少目标规格或当前组件上下文时，不允许凭感觉重写整页；应先补截图、DOM、组件或样式上下文。

Electron 视觉任务的 First-Shot Design Pack 还必须明确：验收以 Electron 真窗口为准，不能只用网页端截图代替。

1. 目标提取
   - 从用户图片、Figma、截图或文字需求中提取目标。
   - 保留原图引用，避免只靠摘要。
   - 尽可能提取颜色、字号、间距、布局、组件状态等结构化信息。

2. 预览启动
   - 自动识别项目启动命令。
   - Web 项目启动浏览器预览。
   - Electron 项目启动真窗口。

3. 当前截图
   - 对当前实现截图。
   - 保存截图与目标图关联。

4. 视觉比对
   - 比较目标图和当前截图。
   - 输出结构化差异：
     - 布局偏差
     - 间距偏差
     - 颜色偏差
     - 字体偏差
     - 组件状态偏差
     - 图标/图片缺失
     - 内容溢出或遮挡

5. 精准修复
   - 把差异清单作为下一轮 Agent 输入。
   - 要求小步修复，不重新生成整页。

6. 再截图验证
   - 修复后再次截图。
   - 至少执行一轮视觉复查。
   - 如果差异仍大，继续循环直到达到阈值或达到最大轮次。

7. 结果呈现
   - Trace Viewer 展示目标图、当前图、差异清单、修复动作和最终状态。

## 循环控制

默认建议：

1. `dev` 模式最多自动修复 3 轮。
2. `visual-dev` 模式最多自动视觉修复 3 轮。
3. 每轮必须有明确证据：
   - 命令输出
   - 截图
   - 差异清单
   - 修复摘要
4. 如果连续两轮差异没有明显改善，暂停并提示用户确认方向。
5. 如果涉及破坏性操作、登录态、支付、生产环境等风险，暂停等待确认。

## UI 设计

### 输入区

在 Prompt 输入区增加一个自动状态：

- `Dev Loop: 自动`
- `视觉验证: 自动`
- `真窗口验证: 自动`

状态可以折叠显示，不干扰普通聊天。

### 执行轨迹

Trace Viewer 增加 Dev Loop 相关节点：

1. `任务分类`
2. `启动预览`
3. `截图当前页面`
4. `视觉比对`
5. `修复偏差`
6. `再次验证`
7. `完成证据`

### 视觉比对详情

右侧详情面板展示：

1. 目标图
2. 当前截图
3. 差异清单
4. 置信度/严重程度
5. 对应修复 commit 或文件修改
6. 下一轮建议

## 数据模型建议

新增或扩展事件模型：

```ts
type DevLoopMode = "none" | "dev" | "visual-dev" | "electron-window";

type DevLoopEvent = {
  type: "dev_loop";
  mode: DevLoopMode;
  phase:
    | "classified"
    | "preview_started"
    | "screenshot_captured"
    | "visual_compared"
    | "fix_applied"
    | "verified"
    | "paused"
    | "completed";
  iteration: number;
  summary: string;
  artifacts?: Array<{
    kind: "target" | "screenshot" | "diff" | "log";
    uri: string;
    label: string;
  }>;
  diagnostics?: Array<{
    category: "layout" | "spacing" | "color" | "typography" | "content" | "state" | "other";
    severity: "low" | "medium" | "high";
    detail: string;
    suggestion?: string;
  }>;
};
```

## MVP 范围

第一版只做最小闭环：

1. 开发任务分类：
   - 识别 `code/frontend/visual/electron/docs`。
   - 给会话打上 `loopMode`。

2. Prompt 注入：
   - 对 `dev` 任务自动要求运行验证。
   - 对 `visual-dev` 任务自动要求截图和视觉比对。

3. Trace 展示：
   - 把 Dev Loop 分类、验证命令、截图、差异清单作为节点展示。

4. 视觉任务最少自动执行一轮：
   - 生成代码。
   - 启动预览。
   - 截图。
   - 产出差异清单。
   - 修复一轮。

## 后续增强

1. 多轮自动视觉修复，支持阈值判断。
2. Figma API 自动拉取 frame、token、截图。
3. 浏览器 DOM/CSS 采样，减少纯视觉猜测。
4. 目标图与当前图 overlay 差异视图。
5. 对每个项目记忆启动命令和预览 URL。
6. 对视觉修复生成专门的 patch prompt，避免整页重写。

## 验收标准

1. 用户提交普通开发任务时，系统默认提示并执行验证闭环。
2. 用户提交图片/Figma/UI 任务时，系统默认进入视觉开发闭环。
3. 用户不需要手动说“截图验证”也能看到截图和差异清单。
4. Trace Viewer 能回放每一轮：目标、当前截图、差异、修复、验证结果。
5. 简单非开发问答不会误触发 Dev Loop。
6. 用户可以手动关闭或降级闭环。

## 自检

1. 设计聚焦 `tech-cc-hub` 默认开发体验，不扩散到模型替换。
2. 通用开发闭环和视觉开发闭环边界清晰。
3. 明确了默认启用条件、降级条件和最大循环控制。
4. MVP 可分阶段落地，不要求一次实现全部 Cursor 体验。
5. Trace Viewer 和现有会话/事件模型有清晰衔接点。

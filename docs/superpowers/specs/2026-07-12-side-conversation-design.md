# 临时多线程 BTW 侧聊设计

## 目标

在右侧 Activity Rail 提供类似 Claude Code `/btw` 的临时侧聊，但扩展为多轮、多个线程并允许完整工具调用。侧聊继承主会话创建时的上下文快照，之后独立发展，不向主会话写回任何消息、工具记录或状态。

## 已确认决策

- 侧聊采用临时多轮模型，不是单次回答。
- 同一个主会话可以同时拥有多个侧聊线程。
- 每个侧聊线程创建时获取主会话上下文快照；主会话后续消息不再同步进该线程。
- 每个侧聊线程拥有完整工具能力，并沿用主会话创建时的工作目录、工具目录和权限策略。
- 左侧主输入框和右侧侧聊输入框同时存在，状态完全分离。
- 两个输入框复用同一个 Composer UI 核心，不共享草稿、附件、引用、队列、焦点或发送控制器。
- 打开第一个侧聊时聊天区域为空，不渲染主会话历史。
- 从文本选区点击“侧边对话”只打开 BTW 侧栏，不自动写入选中文本。
- 关闭单个侧聊线程立即停止执行并清除该线程全部数据。
- 关闭整个“侧聊”页签立即清除当前主会话的全部侧聊线程。
- 侧聊数据不持久化；应用重启不恢复。

## 方案选择

### 采用：独立的内存 BTW Runtime

每个侧聊线程使用独立 runner 和独立临时状态，只存在于内存。它不注册为普通会话，不写入会话数据库，不出现在左侧栏、历史记录、归档或恢复流程中。

该方案可以真正隔离多轮历史、运行状态、停止信号、权限请求和工具结果，同时复用现有 runner、工具和渲染能力。关闭线程时可以确定性销毁全部数据。

### 拒绝：隐藏的普通会话

把侧聊实现成隐藏 session 可以快速复用现有链路，但容易被数据库、会话列表、自动激活、历史恢复和归档逻辑误处理，无法可靠满足“关闭即清除”。

### 拒绝：主会话内的子消息流

在主 runner 内按 thread ID 复用消息流会让停止、权限、工具结果和上下文互相串线，也无法安全并发运行多个侧聊。

## 领域模型

一个主会话可以关联零到多个 `BtwThread`：

```text
MainSession
├─ main runner / main history / main composer
└─ BtwThread[]
   ├─ thread A: snapshot A + private turns + private runner
   ├─ thread B: snapshot B + private turns + private runner
   └─ thread C: snapshot C + private turns + private runner
```

每个 `BtwThread` 至少包含：

- `id`：仅在当前进程有效的唯一 ID。
- `parentSessionId`：所属主会话 ID。
- `title`：默认使用“侧聊 1、侧聊 2……”；可根据首个问题更新显示标题，但不写入普通会话系统。
- `createdAt`、`updatedAt`。
- `status`：`idle | running | completed | error`。
- `messages`：仅该侧聊的用户消息、助手消息和工具结果。
- `draft`、`attachments`、引用、队列与 Composer 局部状态。
- `permissionRequests`：仅该线程的待确认工具权限。
- `error`：仅该线程的运行错误。

主进程的临时 runtime 另外保存：

- 创建时的主会话上下文快照。
- 创建时的 cwd、模型、推理模式、权限模式和允许工具集合。
- 独立 runner/abort handle。
- 该线程后续多轮所需的私有运行历史。

## 上下文快照

创建侧聊线程时，主进程从主会话当前有效上下文构建一次固定快照。快照应复用现有 stateless continuation/上下文压缩路径，覆盖当前主会话已知的系统说明、项目上下文、对话、工具结果和必要的运行元数据，而不是只复制屏幕上可见文本。

快照创建后保持不变：

- 主会话新增消息不会进入既有侧聊。
- 侧聊的用户消息、助手回复和工具结果只追加到该侧聊的私有历史。
- 新建另一个侧聊时重新获取当时最新的主会话快照，因此不同侧聊可以拥有不同的创建基线。

## 工具与权限

侧聊使用完整工具目录，包括文件、命令、浏览器、MCP 和其他当前 runner 支持的工具。工具副作用作用于真实工作区，因此可能与主任务或其他侧聊并发修改同一文件。

隔离规则：

- 每个线程拥有独立工具调用 ID、进度、结果和权限请求。
- 权限策略继承创建时的主会话设置；需要确认的操作仍在对应侧聊中确认。
- 停止一个侧聊只终止该线程，不停止主任务或其他侧聊。
- 工具输出只进入对应侧聊的私有上下文，不进入主会话消息历史。
- 系统不静默串行化多个线程；UI 明确显示各线程运行状态。并发写入冲突由工具结果正常暴露，不伪装成成功。

## 主进程 Runtime 与协议

新增内存 `BtwRuntimeManager`，按 `threadId` 管理临时 runtime。它只复用 runner 能力，不复用普通会话持久化和自动激活流程。

建议使用独立事件协议：

- `btw.thread.create`：从指定主会话创建空白临时线程和上下文快照。
- `btw.thread.send`：向指定线程发送一轮消息。
- `btw.thread.stop`：停止指定线程当前执行。
- `btw.thread.permission.response`：处理该线程权限请求。
- `btw.thread.close`：停止并销毁单个线程。
- `btw.parent.close_all`：停止并销毁某个主会话的所有线程。

服务端事件同样使用 `btw.*` 命名并始终携带 `threadId`，包括状态、用户消息、流式消息、工具进度、权限请求和错误。普通 `session.*` 事件与 Zustand 主会话状态不参与侧聊路由。

## Renderer 状态

新增独立的内存 `useBtwStore`，按 `parentSessionId` 保存线程顺序和当前选中线程，按 `threadId` 保存可渲染消息、草稿、附件、状态、错误和权限请求。

`useAppStore` 继续只管理普通会话。BTW 事件只能写入 `useBtwStore`，禁止调用 `setActiveSessionId`、普通会话消息追加或持久化方法。

切换左侧主会话时，右栏切换到该主会话自己的临时线程集合。未关闭的线程在当前应用进程内保留；删除主会话时立即执行 `close_all`。

## UI 与 Composer 复用

Activity Rail 保留可选“侧聊”页签。打开时：

- 如果当前主会话没有侧聊线程，自动创建“侧聊 1”。
- 新线程聊天区为空，只显示独立输入框。
- 顶部显示多个侧聊标签、运行状态、单线程关闭按钮和新增 `+` 按钮。
- 切换标签只切换右栏 BTW 状态，不改变主 `activeSessionId`。
- 关闭最后一个线程后保留空侧聊页签，并允许再次新建；关闭 Activity Rail 的“侧聊”页签则清除所有线程。

输入组件采用“共享视图、独立控制器”：

- 从当前 `PromptInput` 中提取可复用 Composer UI 核心。
- 主 Composer controller 继续绑定普通会话 store 和 `session.*` 发送链路。
- BTW Composer controller 绑定 `useBtwStore` 和 `btw.*` 发送链路。
- 两个实例同时挂载，分别维护 contentEditable DOM、草稿、附件、引用、队列、Slash 菜单、模型选择和焦点。
- 侧聊模型或权限选择只影响对应线程，不修改主会话设置。

消息区复用现有 `ChatTranscript`、工具过程卡和权限 UI，但传入 BTW 线程消息与 thread-scoped handler。初始消息数组为空，因此不会镜像主聊天内容。

## 生命周期与清理

### 创建

1. Renderer 生成创建请求并显示创建中标签。
2. 主进程验证父会话存在，构建上下文快照和临时 runtime。
3. 返回空消息线程；不触发普通会话创建或激活。

### 发送

1. BTW Composer 将当前线程草稿和附件发送给 `btw.thread.send`。
2. Runtime 使用固定主快照加私有侧聊历史执行下一轮。
3. 状态、工具和输出事件只更新该 thread。

### 关闭单线程

1. 立即禁用该线程交互。
2. abort 正在运行的请求并拒绝后续迟到事件。
3. 销毁 runner、上下文快照和主进程 map 条目。
4. 删除 Renderer 的消息、草稿、附件、引用、队列、权限和错误。
5. 选择相邻线程；没有线程时显示空状态。

### 关闭整个侧聊页签

1. 对当前主会话执行 `btw.parent.close_all`。
2. 清空该主会话在 `useBtwStore` 中的全部线程状态。
3. 切回 Activity Rail 默认页签。

### 其他清理

- 删除父会话时清除其所有侧聊。
- 应用退出时 abort 全部 BTW runtime。
- 进程崩溃或重启后不恢复侧聊。

## 错误处理

- 父会话不存在或快照失败：线程创建失败并显示可关闭错误状态，不创建普通会话兜底。
- 某线程 runner 失败：只标记该线程错误，不影响主任务或其他线程。
- 权限请求在线程关闭后返回：按已销毁 thread ID 丢弃。
- 关闭过程中收到流式迟到事件：RuntimeManager 和 Renderer 都检查 thread 是否仍存在后再处理。
- 工具并发修改冲突：展示真实工具失败或文件状态，不自动回滚其他线程的工作。

## TDD 与验收

实现严格执行 RED-GREEN-REFACTOR：

1. 先测试 BTW store 的多线程隔离、标签切换和关闭清理。
2. 先测试创建时快照固定，主会话后续消息不会进入既有侧聊。
3. 先测试 BTW 协议不会触发普通会话创建、激活、持久化或历史追加。
4. 先测试完整工具、权限、停止和迟到事件按 thread ID 隔离。
5. 先测试主 Composer 与 BTW Composer 同时存在且草稿、附件、引用和焦点分离。
6. 先测试初始聊天区为空、多个标签可切换、关闭单线程和关闭全部会清除数据。
7. 浏览器 QA 覆盖主任务运行时创建两个侧聊、独立多轮、模拟工具结果、关闭一个线程后另一个仍在、关闭页签后全部清空。

完成标准：

- 聚焦测试和相关回归全部通过。
- TypeScript 与生产构建通过。
- 范围 ESLint 零错误，`git diff --check` 通过。
- 浏览器 QA 证明主历史未被污染、双输入框同时存在、多个线程相互隔离、关闭后数据不可恢复。
- visual-verdict 不低于 90/100。

## 非目标

- 不把 BTW 线程保存到数据库、磁盘或会话历史。
- 不在左侧会话栏、归档、恢复或搜索中显示 BTW 线程。
- 不把侧聊消息合并回主会话。
- 不自动同步侧聊创建后的主会话新增内容。
- 不新增第三方依赖。

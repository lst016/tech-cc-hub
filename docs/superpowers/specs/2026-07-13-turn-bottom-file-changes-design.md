# 每轮会话底部统一展示文件修改

## 背景

当前聊天记录按连续的工具消息生成多个过程组，每个过程组自行提取并渲染文件修改卡片。一轮会话如果包含多段过程消息，文件修改卡片会夹在该轮的文字回复或其他过程内容之间，也可能在同一轮重复出现。

## 目标

- 文件修改仍归属于产生修改的用户轮次。
- 同一轮的全部文件修改只汇总展示一次。
- 文件修改卡片只出现在该轮底部：下一轮用户消息之前；最后一轮则位于当前已渲染内容末尾。
- 历史轮次和正在执行的当前轮次遵循同一规则。
- 过程明细、助手文字、生成图片及工作流代理卡片保持原有相对顺序。

## 非目标

- 不改变文件修改的识别、路径解析、增删行统计或预览打开行为。
- 不把所有轮次的文件修改汇总到整场会话最底部。
- 不重排工具过程、助手回复或生成图片。
- 不引入新依赖或修改持久化消息格式。

## 设计

### 轮次边界

以 `user_prompt` 作为新一轮开始。一个轮次从该 `user_prompt` 起，持续到下一个 `user_prompt` 之前；最后一轮持续到当前消息列表末尾。消息窗口从某轮中间开始时，当前可见片段沿用其所属轮次，并在该轮可见底部展示汇总卡片。

### 数据流

1. 构建聊天渲染条目时继续按原规则生成分隔符、普通消息、过程组和工作流代理卡片。
2. 同时收集当前轮次内所有过程消息，供文件修改提取逻辑使用。
3. 遇到下一条 `user_prompt` 前，先在上一轮末尾追加一个轮次文件修改条目；遍历结束时为最后一轮执行同样的收尾。
4. 轮次文件修改条目复用现有文件变更提取、去重、统计和预览交互。
5. 过程组继续渲染过程明细和生成图片，但不再自行渲染文件修改卡片。

主聊天和复用 `ChatTranscript` 的侧边会话、工作流代理会话应共用同一轮次收尾规则，避免同一份历史记录在不同入口出现不同顺序。

### 展示行为

- 一轮没有文件修改时，不生成空卡片。
- 同一文件在一轮内被多次修改时，沿用现有去重与累计统计逻辑。
- 多轮分别展示各自的修改卡片，不跨轮合并。
- 当前轮正在执行时，卡片始终跟随该轮当前内容末尾；新过程或回复到达后，卡片重新落到最新内容之后。
- 点击文件仍打开右侧预览并定位首个修改。

## 实现边界

- 将文件修改卡片从过程组的渲染职责中拆出为可复用的轮次级组件。
- 将轮次收尾与文件修改条目的生成放到可测试的纯函数中，供主聊天和共享聊天记录组件调用。
- 保留过程组现有展开、分批显示、生成图片和消息锚点行为。
- 修改保持局部，不重构无关聊天或消息存储代码。

## 验证

新增回归测试覆盖：

1. 同一轮包含多段过程组时，仅在该轮末尾生成一个文件修改条目。
2. 文件修改之后还有助手文字时，修改条目位于文字之后。
3. 连续多轮分别在各自底部展示，不跨轮合并。
4. 无文件修改的轮次不生成修改条目。
5. 当前最后一轮的修改条目位于当前渲染列表末尾。
6. 现有文件预览路由、过程组布局和生成图片测试继续通过。

验证命令以仓库现有 Electron 测试编译和相关定向测试为准，并补充 TypeScript 检查与 `git diff --check`。

## 风险与缓解

- **消息窗口从轮次中间开始：** 使用原始消息索引和当前可见轮次收尾，避免卡片被放到错误的下一轮。
- **主聊天与共享聊天记录逻辑漂移：** 抽取同一个纯函数生成轮次级条目。
- **过程组功能回归：** 文件卡片以外的过程组内容不重排，并运行现有布局、图片和预览路由测试。

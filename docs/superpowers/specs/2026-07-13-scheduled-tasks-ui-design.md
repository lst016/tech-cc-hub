# 定时任务 UI 批量优化设计

## 目标

让用户进入页面后能快速回答四个问题：有哪些任务、当前是否正常、何时再次执行、下一步可以做什么。创建和编辑任务时，应先理解任务内容与运行位置，再配置调度计划，并始终看到可读的计划摘要。

## 现状证据

- 页面头部重复出现“定时任务”，主操作仅为弱描边按钮。
- 空状态只有一行提示和大面积虚线框，缺少解释与就地创建入口。
- 当前列表容器存在无效的 `max-w-800px`，高信息量卡片却在宽屏排三列。
- 卡片没有键盘语义、选中反馈和清晰操作分层；详情远离被选任务。
- 创建弹窗把内容、会话、工作区和调度配置平铺为一条长表单，首屏会截断时间设置。
- 状态标签使用原始颜色而非项目语义 token。

现状截图保存在 `.omx/state/cron-ui-audit/01-before-empty.png` 与 `.omx/state/cron-ui-audit/02-before-create-dialog.png`。

## 方案比较

### A. 任务看板 + 同屏详情（采用）

宽屏使用两列任务卡片，卡片承担扫描，点击后在当前列表下方展开详情。顶部提供紧凑统计、搜索和状态筛选。优点是保留工作区分组、适合桌面工作台，也能在不新增路由的情况下提升信息层级。

### B. 密集表格

使用任务名、状态、计划、下次运行、工作区和操作列。扫描效率高，但长描述、移动宽度和当前内联详情都不适合表格，容易把产品做成通用后台。

### C. 左侧列表 + 右侧详情

选择任务后在右侧固定详情面板。上下文最稳定，但当前应用已有右侧工作区，页面可用宽度不稳定，容易造成嵌套侧栏。

## 采用设计

### 页面

- 内容宽度使用 `max-w-[1040px]`，页面以 warm utilitarian workbench 为基准。
- 头部只保留一处标题，使用 clay accent 主按钮。
- 增加总任务、运行中、已暂停、异常四项紧凑统计。
- 增加搜索和状态筛选；筛选不改变后端数据。
- 加载态使用任务卡骨架；错误态明确说明并提供重试；空态提供图标、说明和创建入口。

### 任务卡与详情

- 宽屏两列、窄屏单列；每张卡片按标题/描述、状态、调度、下次运行、工作区组织。
- 卡片使用语义按钮并支持键盘；选中态使用 `accent-subtle` 和 `accent/25`。
- 状态颜色改用 `success`、`warning`、`error`、`surface-secondary` token。
- 卡片只保留启停开关和更多菜单；编辑、绑定和删除放在菜单与详情。
- 详情靠近所选任务，立即运行、编辑、启停、绑定、删除保持一致；删除统一经过确认。

### 创建与编辑

- 弹窗宽度扩展到 720px，内容分为“任务内容、运行位置、调度计划”。
- 输入控件统一使用项目 surface、border、focus ring 和圆角。
- 执行方式改为可读的分段选择；工作区与频率仍使用原生 select 保持稳定。
- 调度区增加实时摘要，使用户保存前看到“每天 09:00 执行”之类的结果。
- 不修改 cron 数据模型、IPC 合约或调度语义。

## 回归保护与验证

- 浏览器预览增加 `qaCron=populated|empty|error` fixture，支持列表、空态、错误态和弹窗截图。
- 增加源码契约测试，保护语义状态、键盘/ARIA、确认删除和 fixture。
- 运行聚焦测试、TypeScript/build、定时任务 smoke、响应式截图和 console 检查。
- 每轮截图执行 visual verdict；低于 90 分继续修改。

## 非目标

- 不新增任务执行历史、导入导出、通知或新的后端能力。
- 不引入新依赖或通用组件库。
- 不重构 cron service、repository 或数据库。

## 清理计划

1. 先用测试锁住现有 IPC 与调度行为。
2. 删除页面中的无效 Tailwind 类、重复标题、重复操作和手写 SVG。
3. 复用 `AppModalOverlay`、现有 token 与 `lucide-react`，只保留局部展示组件。
4. 将视觉改动限制在 cron renderer 与 QA fixture，避免触碰无关脏改。
